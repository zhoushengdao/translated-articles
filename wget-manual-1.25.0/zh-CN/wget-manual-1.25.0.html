1 概述 ¶

GNU Wget 是一个用于从网络非交互式下载文件的免费工具。 它支持 HTTP、HTTPS 和 FTP 协议，以及通过 HTTP 代理的检索。

本章是 Wget 功能的部分概述。

  • Wget 是非交互式的，这意味着它可以在后台工作，而用户未登录。 这允许您开始检索并从系统断开连接，让 Wget 完成工作。 相比之下，大多数网络浏览器需要用户的持续存在，这在传输大量数据时可能是一个很大的障碍。
  • Wget 可以跟随 HTML、XHTML 和 CSS 页面中的链接，创建远程网站的本地版本，完全重现原始网站的目录结构。 这有时被称为“递归下载”。在执行此操作时，Wget 尊重机器人排除标准（/robots.txt）。 可以指示 Wget 将下载文件中的链接转换为指向本地文件，以便离线查看。
  • 文件名通配符匹配和目录的递归镜像在通过 FTP 检索时可用。 Wget 可以读取 HTTP 和 FTP 服务器提供的时间戳信息，并本地存储。 因此，Wget 可以查看远程文件自上次检索以来是否已更改，并在已更改时自动检索新版本。 这使得 Wget 适合镜像 FTP 站点以及主页。
  • Wget 设计用于在缓慢或不稳定的网络连接上保持稳健；如果下载因网络问题失败，它将不断重试，直到整个文件被检索。 如果服务器支持重新获取，它将指示服务器从中断处继续下载。
  • Wget 支持代理服务器，可以减轻网络负载，加快检索速度，并提供防火墙后的访问。 Wget 默认使用被动 FTP 下载，主动 FTP 是一个选项。
  • Wget 支持下一代 IP 标准 IPv6。 IPv6 在编译时自动检测，并且可以在构建或运行时禁用。 构建时支持 IPv6 的二进制文件在仅 IPv4 和双栈环境中都能很好地工作。
  • 内置功能提供了调整您希望跟随的链接的机制（参见 跟随链接）。
  • 使用进度条跟踪单个下载的进度。 交互式下载使用“温度计”式进度条跟踪，而非交互式下载则用点跟踪，每个点代表接收到的固定数据量（默认为1KB）。 可以根据您的偏好自定义任一进度条。
  • 大多数功能都可以完全配置，无论是通过命令行选项，还是通过初始化文件 .wgetrc（参见 启动文件）。 Wget 允许您定义全局启动文件（默认为 /usr/local/etc/wgetrc）以进行站点设置。 您还可以使用 –config 选项指定启动文件的位置。 要禁用配置文件的读取，请使用 –no-config。 如果同时给出 –config 和 –no-config，则忽略 –no-config。
  • 最后，GNU Wget 是自由软件。 这意味着每个人都可以使用它，重新分发它和/或在 GNU 通用公共许可证的条款下修改它，由自由软件基金会发布（参见随 GNU Wget 附带的文件 COPYING 以获取详细信息）。

2 调用 ¶

默认情况下，Wget 的调用非常简单。 基本语法是： 

wget [选项]…… [URL]……

Wget 将简单地下载命令行中指定的所有 URL。 URL 是 统一资源定位符，如下定义。

然而，您可能希望更改 Wget 的一些默认参数。 您可以通过两种方式做到这一点：永久性地，将适当的命令添加到 .wgetrc（参见 启动文件），或在命令行上指定它。

2.1 URL 格式 ¶

URL 是统一资源定位符的缩写。 统一资源定位符是通过互联网可用的资源的紧凑字符串表示。 Wget 识别按照 RFC1738 标准定义的 URL 语法。 这是最广泛使用的形式（方括号表示可选部分）： 

http://host[:port]/directory/file
ftp://host[:port]/directory/file

您也可以在 URL 中编码您的用户名和密码： 

ftp://user:password@host/path
http://user:password@host/path

user 或 password 两者都可以省略。 如果你省略 HTTP 用户名或密码，将不会发送认证信息。 如果你省略 FTP 用户名，将使用 ‘anonymous’。 如果你省略 FTP 密码，你的电子邮件地址将作为默认密码提供。1

重要提示：如果你在命令行上指定包含密码的 URL，用户名和密码将通过 ps 对所有系统用户可见。 在多用户系统上，这是一个很大的安全风险。 要解决这个问题，使用 wget -i - 并将 URL 输入到 Wget 的标准输入，每个 <1>URL 一行，以 C-d 结束。

你可以在 URL 中将不安全字符编码为 ‘%xy’，xy 是字符的 ASCII 值的十六进制表示。 一些常见的不安全字符包括 ‘%’（编码为 ‘%25’），‘:’（编码为 ‘%3A’），和 ‘@’（编码为 ‘%40’）。 有关不安全字符的完整列表，请参阅 RFC1738。

Wget 还支持 FTP URL 的 type 功能。 默认情况下，FTP 文档以二进制模式（type ‘i’）检索，这意味着它们将原封不动地下载。 另一个有用的模式是 ‘a’（ASCII）模式，它转换不同操作系统之间的行分隔符，因此对文本文件很有用。 这是一个例子： 

ftp://host/directory/file;type=a

由于安全考虑，不再支持 URL 规范的两个替代变体：

FTP-only 语法（由 NcFTP 支持）： 

host:/dir/file

HTTP-only 语法（由Netscape引入）： 

host[:port]/dir/file

这两种替代形式早已被弃用，并且在 1.22.0 版本中移除了支持。

2.2 选项语法 ¶

因为 Wget 使用 GNU getopt 处理命令行参数，所以每个选项都有一个长格式和短格式。 长选项更容易记住，但需要更多时间输入。 你可以自由地混合不同的选项风格，或在命令行参数后指定选项。 因此你可以写成： 

wget -r --tries=10 http://fly.srk.fer.hr/ -o log

接受参数的选项和参数之间的空格可以省略。 你可以写成‘-olog’而不是‘-o log’。

你可以将几个不需要参数的选项放在一起，像这样： 

wget -drc URL

这完全等同于： 

wget -d -r -c URL

由于选项可以在参数后指定，你可以用‘--’来终止它们。 所以以下命令将尝试下载URL ‘-x’，并将失败报告到log： 

wget -o log -- -x

接受逗号分隔列表的选项都遵循指定空列表清除其值的约定。 这对于清除.wgetrc设置很有用。 例如，如果你的.wgetrc将exclude_directories设置为/cgi-bin，以下示例将首先重置它，然后将其设置为排除/~nobody和/~somebody。 你也可以清除.wgetrc中的列表 （参见Wgetrc语法）。 

wget -X "" -X /~nobody,/~somebody

大多数不接受参数的选项是布尔选项，之所以这样命名是因为它们的状态可以用是或否 （“布尔”）变量来捕获。 例如，‘--follow-ftp’告诉 Wget 从 HTML 文件中跟随 FTP 链接，另一方面，‘--no-glob’告诉它不要在 FTP URL 上执行文件通配。 一个布尔选项要么是肯定的要么是否定的（以‘--no’开头）。 所有这些选项都有几个共同属性。

除非另有说明，否则假定默认行为是选项所完成的操作的反面。 例如，‘--follow-ftp’的文档存在假定默认是不从 HTML 页面跟随 FTP 链接。

肯定选项可以通过在选项名称前加上‘--no-’来否定；否定选项可以通过省略‘--no-’前缀来否定。 这似乎多余—如果肯定选项的默认行为是不做某事，那么为什么提供一种方式来明确关闭它？ 但启动文件实际上可能改变默认值。 例如，在.wgetrc中使用 follow_ftp = on 使 Wget 默认跟随 FTP 链接， 使用‘--no-follow-ftp’是从命令行恢复默认的唯一方式。

2.4 日志和输入文件选项 ¶

‘-o logfile’ ¶
‘--output-file=logfile’

将所有消息记录到 logfile。 消息通常报告给标准错误。

‘-a logfile’ ¶
‘--append-output=logfile’

追加到 logfile。 这与‘-o’相同，只是它追加到 logfile 而不是覆盖旧的日志文件。 如果 logfile 不存在，将创建一个新文件。

‘-d’ ¶
‘--debug’

开启调试输出，这意味着如果 Wget 不能正常工作，对 Wget 开发者来说非常重要的各种信息将会输出。 你的系统管理员可能选择在没有调试支持的情况下编译 Wget，在这种情况下，‘-d’将不起作用。 请注意，使用调试支持编译总是安全的—带有调试支持的 Wget 编译将不会打印任何调试信息，除非使用‘-d’选项。 参见报告错误，了解更多关于如何使用‘-d’发送错误报告的信息。

‘-q’ ¶
‘--quiet’

关闭 Wget 的输出。

‘-v’ ¶
‘--verbose’

开启详细输出，包含所有可用数据。 默认输出是详细的。

‘-nv’
‘--no-verbose’

关闭详细输出而不完全静默（使用‘-q’来实现），这意味着错误消息和基本信息仍然会被打印。

‘--report-speed=type’

以type输出带宽。 唯一接受的值是‘bits’。

‘-i file’ ¶
‘--input-file=file’

从本地或外部文件读取 URL。 如果指定‘-’为文件，则从标准输入读取 URL。 （使用‘./-’从字面上名为‘-’的文件中读取。）

如果使用此功能，则命令行上不需要存在 URL。 如果命令行和输入文件中都有 URL，则命令行上的 <0>URL 将首先被检索。 如果未指定‘--force-html’，则文件应包含一系列 URL，每行一个。

但是，如果指定了‘--force-html’，文档将被视为‘html’。 在这种情况下，你可能会遇到相对链接的问题，你可以通过在文档中添加 <base href="url"> 或在命令行上指定‘--base=url’来解决。

如果文件是外部的，当 Content-Type 匹配‘text/html’时，文档将自动被视为‘html’。 此外，如果未指定，文件的位置将隐式用作基本href。

如果文件是本地文件，在支持的系统上，它将使用 O_NONBLOCK 标志打开，以允许从文件中进行非阻塞读取。 ‘Wget’ 将尝试继续从文件中读取，直到达到 EOF 或文件关闭。 这允许在进程执行期间动态生成的文件列表进行流式检索。 目前，此功能在Windows平台上不可用。

‘--input-metalink=文件’ ¶

下载本地Metalink文件中涵盖的文件。 支持Metalink版本3和4。

‘--keep-badhash’ ¶

保留下载的Metalink文件，即使其哈希值不正确。 它将.badhash附加到具有校验和不匹配的Metalink文件的名称上，除非不覆盖现有文件。

‘--metalink-over-http’ ¶

发出HTTP HEAD请求而不是GET，并从响应头中提取Metalink元数据。 然后切换到Metalink下载。 如果未找到有效的Metalink元数据，则回退到普通HTTP下载。 启用‘Content-Type: application/metalink4+xml’文件的下载/处理。

‘--metalink-index=数字’ ¶

设置Metalink ‘application/metalink4+xml’ metaurl序号NUMBER。 从1到可用的“application/metalink4+xml”总数。 指定0或‘inf’以选择第一个好的。 Metaurls，例如来自‘--metalink-over-http’的，可能已按优先级键的值排序；记住这一点以选择正确的NUMBER。

‘--preferred-location’ ¶

设置Metalink资源的首选位置。 如果有多个具有相同优先级的资源可用，则此设置有效。

‘--xattr’ ¶

启用使用文件系统的扩展属性来保存原始URL和Referer HTTP头值（如果使用）。

请注意，URL可能包含私人信息，如访问令牌或凭据。

‘-F’ ¶
‘--force-html’

当从文件读取输入时，强制将其视为HTML文件。 这使您能够通过添加<base href="url">到HTML，或使用‘--base’命令行选项，从本地磁盘上的现有HTML文件中检索相对链接。

‘-B URL’ ¶
‘--base=URL’

当通过‘-i’/‘--input-file’选项（与‘--force-html’一起，或当输入文件从服务器远程获取并描述为HTML时）从HTML文件读取链接时，使用URL作为参考点解析相对链接。 这相当于在HTML输入文件中存在BASE标签，并以URL作为href属性的值。

例如，如果您为URL指定‘http://foo/bar/a.html’，并且Wget从输入文件中读取‘../baz/b.html’，它将被解析为‘http://foo/baz/b.html’。

‘--config=FILE’ ¶

指定您希望使用的启动文件的位置，而不是默认文件。 使用–no-config来禁用配置文件的读取。 如果同时给出–config和–no-config，则忽略–no-config。

‘--rejected-log=logfile’

将所有URL拒绝记录到logfile中，作为逗号分隔的值。 这些值包括拒绝的原因、URL以及找到它的父URL。

2.5 下载选项 ¶

‘--bind-address=ADDRESS’ ¶

当建立客户端TCP/IP连接时，绑定到本地机器上的ADDRESS。 ADDRESS可以指定为主机名或IP地址。 如果您的机器绑定到多个IP，此选项可能有用。

‘--bind-dns-address=ADDRESS’ ¶

[仅libcares] 此地址覆盖DNS请求的路由。 如果您需要绕过/etc/resolv.conf中的标准设置，此选项与‘--dns-servers’是您的朋友。 ADDRESS必须指定为IPv4或IPv6地址。 Wget需要与libcares一起构建才能使用此选项。

‘--dns-servers=ADDRESSES’ ¶

[仅libcares] 给定的地址(es)覆盖标准名称服务器地址，例如在/etc/resolv.conf中配置的地址。 ADDRESSES可以指定为IPv4或IPv6地址，逗号分隔。 Wget需要与libcares一起构建才能使用此选项。

‘-t number’ ¶
‘--tries=number’

将尝试次数设置为number。 指定0或‘inf’进行无限重试。 默认是重试20次，除了像“连接被拒绝”或“未找到”（404）这样的致命错误，这些错误不会被重试。

‘-O file’
‘--output-document=file’

文档将不会写入适当的文件，而是全部连接在一起并写入file。 如果使用‘-’作为file，文档将被打印到标准输出，禁用链接转换。 (使用‘./-’打印到字面名为‘-’的文件。)

使用‘-O’并不意味着简单地使用名称file代替URL中的名称；相反，它类似于shell重定向：‘wget -O file http://foo’旨在像‘wget -O - http://foo > file’一样工作；file将立即被截断，所有下载的内容都将写入那里。

因此，‘-N’（用于时间戳检查）不支持与‘-O’结合使用：因为file总是新创建的，它将总是有一个非常新的时间戳。 如果使用此组合，将发出警告。

同样，使用‘-r’或‘-p’与‘-O’可能不会如您所愿：Wget不会只是将第一个文件下载到file，然后将其余文件下载到它们的正常名称：所有下载的内容都将放在file中。 这在1.11版本中被禁用，但在1.11.2版本中已恢复（带有警告），因为在某些情况下，这种行为实际上可能有一些用途。

只有当给定的输出文件不存在时，才接受与‘-nc’的组合。

请注意，只有当下载单个文档时，才允许与‘-k’的组合，因为在这种情况下，它只会将所有相对URI转换为外部URI；当所有URI都被下载到单个文件时，‘-k’对于多个URI没有意义；只有当输出是常规文件时，才能使用‘-k’。

‘-nc’ ¶
‘--no-clobber’

如果在同一目录中多次下载文件，Wget的行为取决于几个选项，包括‘-nc’。 在某些情况下，本地文件将在重复下载时被覆盖。 在其他情况下，它将被保留。

当运行Wget而不使用‘-N’、‘-nc’、‘-r’或‘-p’时，在同一目录中下载相同的文件将导致file的原始副本被保留，第二个副本被命名为‘file.1’。 如果再次下载该文件，第三个副本将被命名为‘file.2’，依此类推。 （这也是使用‘-nd’时的行为，即使‘-r’或‘-p’有效。） 当指定‘-nc’时，此行为被抑制，Wget将拒绝下载‘file’的新副本。 因此，“no-clobber”在这种模式下实际上是一个误称——它防止的不是覆盖（因为数字后缀已经防止了覆盖），而是防止了多版本保存。

当运行Wget时使用‘-r’或‘-p’，但不使用‘-N’、‘-nd’或‘-nc’时，重新下载文件将导致新副本简单地覆盖旧副本。 添加‘-nc’将防止这种行为，而是导致保留原始版本，并忽略服务器上的任何新副本。

当运行Wget时使用‘-N’，无论是否使用‘-r’或‘-p’，是否下载文件的新副本取决于文件的本地和远程时间戳和大小（参见时间戳）。 ‘-nc’不能与‘-N’同时指定。

只有当给定的输出文件不存在时，才接受与‘-O’/‘--output-document’的组合。

请注意，当指定‘-nc’时，带有后缀‘.html’或‘.htm’的文件将从本地磁盘加载并解析，就像它们是从Web检索的一样。

‘--backups=backups’ ¶

在（覆盖）写入文件之前，通过向文件名添加‘.1’后缀（在VMS上为‘_1’）来备份现有文件。 这样的备份文件将旋转到‘.2’、‘.3’等，直到backups（并且在那之后丢失）。

‘--no-netrc’ ¶

不要尝试从.netrc文件获取凭据。 默认情况下，如果在命令行上没有传递凭据并且需要身份验证，则会搜索.netrc文件以获取凭据。

‘-c’ ¶
‘--continue’；

继续获取部分下载的文件。 当您想要完成由Wget的先前实例或其他程序启动的下载时，这非常有用。 例如： 

wget -c ftp://sunsite.doc.ic.ac.uk/ls-lR.Z

如果当前目录中有一个名为ls-lR.Z的文件，Wget将假定它是远程文件的第一部分，并将要求服务器从等于本地文件长度的偏移量继续检索。

请注意，如果您只是希望Wget的当前调用在连接中途丢失时重试下载文件，则不需要指定此选项。 这是默认行为。 ‘-c’仅影响在此次Wget调用之前启动的下载的恢复，并且其本地文件仍然存在。

如果没有‘-c’，前面的例子只会将远程文件下载到ls-lR.Z.1，而不会影响截断的ls-lR.Z文件。

如果您在非空文件上使用‘-c’，而服务器不支持继续下载，Wget将从头开始下载并完全覆盖现有文件。

从Wget 1.7开始，如果您在文件大小与服务器上的文件大小相同的文件上使用‘-c’，Wget将拒绝下载该文件并打印解释性消息。 当服务器上的文件比本地文件小（可能是因为自上次下载尝试以来服务器上的文件已更改）时，也会发生同样的情况——因为“继续”没有意义，所以不会发生下载。

另一方面，在使用‘-c’时，任何在服务器上比本地大的文件都将被视为不完整的下载，并且只会下载(length(remote) - length(local))字节并将其附加到本地文件的末尾。 在某些情况下，这种行为可能是可取的——例如，您可以使用‘wget -c’仅下载已附加到数据收集或日志文件的新部分。

然而，如果文件在服务器上更大是因为它被更改了，而不仅仅是附加了，您最终会得到一个混乱的文件。 Wget无法验证本地文件是否确实是远程文件的有效前缀。 当您将‘-c’与‘-r’结合使用时，您需要特别小心这一点，因为每个文件都将被视为“不完整下载”的候选者。

如果您尝试使用‘-c’，另一个您会得到混乱文件的情况是，如果您有一个蹩脚的HTTP代理，它将“传输中断”字符串插入到本地文件中。 未来可能会添加“回滚”选项来处理这种情况。

请注意，‘-c’仅适用于支持Range头的FTP服务器和HTTP服务器。

‘--start-pos=OFFSET’ ¶

从零基位置OFFSET开始下载。 偏移量可以用字节表示，用‘k’后缀表示千字节，用‘m’后缀表示兆字节，等等。

‘--start-pos’的优先级高于‘--continue’。 当同时指定‘--start-pos’和‘--continue’时，wget将发出警告，然后继续操作，就像‘--continue’不存在一样。

需要服务器支持继续下载，否则‘--start-pos’无法帮助。 有关详细信息，请参见‘-c’。

‘--progress=type’ ¶

选择您希望使用的进度指示器类型。 合法的指示器是“dot”和“bar”；。

默认使用“bar”指示器。 它绘制一个ASCII进度条图形（也称为“温度计”显示）指示检索的状态。 如果输出不是TTY，则默认使用“dot”条。

使用‘--progress=dot’切换到“dot”显示。 它通过在屏幕上打印点来跟踪检索，每个点代表固定数量的下载数据。

进度type也可以接受一个或多个参数。 参数根据选择的type而变化。 参数通过附加到类型后，用冒号（:）分隔，如下所示： ‘--progress=type:parameter1:parameter2’。

使用点状检索时，可以通过指定类型为‘dot:style’来设置样式。 不同的样式为一个点赋予不同的意义。 使用default样式时，每个点代表1K，一个簇中有十个点，一行中有五十个点。 binary样式有更“计算机”化的方向——8K点，16点簇和每行48点（每行384K）。 mega样式适合下载大文件——每个点代表64K，一个簇中有八个点，每行有48个点（所以每行包含3M）。 如果mega不够，那么可以使用giga样式——每个点代表1M，一个簇中有八个点，每行有32个点（所以每行包含32M）。

使用‘--progress=bar’时，目前有两个可能的参数，force和noscroll。

当输出不是TTY时，进度条总是回退到“点”，即使‘--progress=bar’在调用时传递给Wget。 这种行为可以通过使用“force”参数作为‘--progress=bar:force’来覆盖并强制“bar”输出。

默认情况下，‘bar’样式的进度条会从左到右滚动正在下载的文件名，如果文件名超过了为其显示分配的最大长度。 在某些情况下，比如使用‘--progress=bar:force’时，可能不希望进度条中的文件名滚动。 通过传递“noscroll”参数，可以强制Wget尽可能显示文件名而不滚动。

注意，你可以使用progress命令在.wgetrc中设置默认样式。 该设置可以从命令行覆盖。 例如，要强制显示条状输出而不滚动，使用‘--progress=bar:force:noscroll’。

‘--show-progress’

强制wget在任何详细模式下显示进度条。

默认情况下，wget仅在详细模式下显示进度条。 然而，有时可能希望wget在屏幕上显示进度条，同时使用其他详细模式，如‘--no-verbose’或‘--quiet’。 这在调用wget下载多个小/大文件时通常是一个期望的属性。 在这种情况下，可以简单地使用此参数调用wget，以获得更清晰的屏幕输出。

此选项还会强制进度条在使用‘--output-file’选项时打印到stderr。

‘--timestamping’

开启时间戳。 详情请参见时间戳。

‘--no-if-modified-since’

在‘-N’模式下不发送If-Modified-Since头。 改为发送初步的HEAD请求。 这仅在‘-N’模式下有效。

‘--no-use-server-timestamps’

不要根据服务器上的时间戳设置本地文件的时间戳。

默认情况下，下载文件时，其时间戳设置为与远程文件匹配。 这允许在后续调用wget时使用‘--timestamping’。 然而，有时基于文件实际下载时间设置本地文件的时间戳很有用；为此，提供了‘--no-use-server-timestamps’选项。

‘-S’ ¶
‘--server-response’

打印HTTP服务器发送的头和FTP服务器发送的响应。

‘--spider’ ¶

当使用此选项调用时，Wget 将表现得像一个网络蜘蛛，这意味着它不会下载页面，只会检查它们是否存在。 例如，您可以使用 Wget 来检查您的书签： 

wget --spider --force-html -i bookmarks.html

为了使 Wget 接近真正的网络蜘蛛的功能，这一特性还需要更多的工作。

‘-T seconds’ ¶
‘--timeout=seconds’

将网络超时设置为seconds秒。 这相当于同时指定‘--dns-timeout’、‘--connect-timeout’和‘--read-timeout’。

与网络交互时，Wget 可以检查超时，如果操作耗时过长则中止操作。 这可以防止像挂起的读取和无限连接这样的异常情况。 默认情况下，唯一启用的超时是900秒的读取超时。 将超时设置为0将完全禁用它。 除非您知道自己在做什么，否则最好不要更改默认的超时设置。

所有与超时相关的选项都接受十进制值，以及亚秒值。 例如，‘0.1’秒是一个合法（虽然不明智）的超时选择。 亚秒超时对于检查服务器响应时间或测试网络延迟非常有用。

‘--dns-timeout=seconds’ ¶

将DNS查找超时设置为seconds秒。 未在指定时间内完成的DNS查找将失败。 默认情况下，除了系统库实现的超时外，DNS查找没有超时。

‘--connect-timeout=seconds’ ¶

将连接超时设置为seconds秒。 建立时间过长的TCP连接将被中止。 默认情况下，除了系统库实现的超时外，没有连接超时。

‘--read-timeout=seconds’ ¶

将读取（和写入）超时设置为seconds秒。 此超时的“时间”指的是空闲时间：如果在下载的任何时候，超过指定秒数没有接收到数据，读取将失败并重新开始下载。 此选项不会直接影响整个下载的持续时间。

当然，远程服务器可能会选择比此选项要求更早地终止连接。 默认的读取超时是900秒。

‘--limit-rate=amount’ ¶

将下载速度限制为每秒amount字节。 数量可以用字节表示，也可以用‘k’后缀表示千字节，或用‘m’后缀表示兆字节。 例如，‘--limit-rate=20k’将限制检索速率为20KB/s。 当您不希望 Wget 消耗所有可用带宽时，此选项非常有用。

此选项允许使用十进制数字，通常与功率后缀一起使用；例如，‘--limit-rate=2.5k’是一个合法值。

请注意，Wget 通过在网络读取后睡眠适当的时间来实现限速，该读取所花费的时间少于速率指定的时间。 最终，这种策略会导致 TCP 传输速度减慢到大约指定的速率。 然而，达到这种平衡可能需要一些时间，所以如果限速对非常小的文件效果不佳，请不要感到惊讶。

‘-w seconds’ ¶
‘--wait=seconds’

在检索之间等待指定的秒数。 建议使用此选项，因为它通过减少请求频率来减轻服务器负载。 时间可以以分钟为单位指定，使用m后缀，以小时为单位使用h后缀，或以天为单位使用d后缀，而不是以秒为单位。

如果网络或目标主机宕机，为此选项指定一个较大的值是有用的，这样Wget可以等待足够长的时间，合理地期望网络错误在重试之前被修复。 此函数指定的等待间隔受--random-wait的影响，请参见。

‘--waitretry=seconds’ ¶

如果你不希望Wget在每次检索之间等待，而仅在失败下载的重试之间等待，你可以使用此选项。 Wget将使用线性回退，在给定文件的第一次失败后等待1秒，然后在第二次失败后等待2秒，直到你指定的最大秒数。

默认情况下，Wget将假定值为10秒。

‘--random-wait’ ¶

一些网站可能会执行日志分析，通过查找请求之间的时间统计显著相似性来识别如Wget这样的检索程序。 此选项使请求之间的时间在0.5到1.5 * wait秒之间变化，其中wait是使用‘--wait’选项指定的，以便从这种分析中掩盖Wget的存在。

2001年，一个致力于流行消费平台开发的出版物中的一篇文章提供了代码来即时执行此分析。 其作者建议在C类地址级别进行封锁，以确保尽管DHCP提供的地址发生变化，自动化检索程序仍被封锁。

‘--random-wait’选项的灵感来自于这个不明智的建议，即由于一个用户的行为而封锁许多不相关的用户访问网站。

‘--no-proxy’ ¶

不使用代理，即使定义了适当的*_proxy环境变量。

参见代理，了解更多关于Wget使用代理的信息。

‘-Q quota’ ¶
‘--quota=quota’

为自动检索指定下载配额。 值可以以字节（默认）、千字节（使用‘k’后缀）或兆字节（使用‘m’后缀）指定。

请注意，配额永远不会影响下载单个文件。 因此，如果你指定‘wget -Q10k https://example.com/ls-lR.gz’，所有的ls-lR.gz都将被下载。 即使命令行上指定了几个URL，情况也是如此。 配额仅在每个下载文件结束时检查，因此它永远不会导致部分下载的文件。 因此，你可以安全地输入‘wget -Q2m -i sites’——在耗尽配额的文件完全下载后，下载将中止。

将配额设置为0或‘inf’将取消下载配额限制。

‘--no-dns-cache’ ¶

关闭DNS查找的缓存。 通常，Wget会记住它从DNS查找的IP地址，这样它就不必为同一（通常很小的）主机集重复联系DNS服务器。 此缓存仅存在于内存中；新的Wget运行将再次联系DNS。

然而，有报道称，在某些情况下，即使对于像Wget这样的短期运行的应用程序，也不希望缓存主机名。 使用此选项，Wget每次建立新连接时都会发出新的DNS查找（更准确地说，是新的gethostbyname或getaddrinfo调用）。 请注意，此选项不会影响可能由解析库或外部缓存层（如NSCD）执行的缓存。

如果你不完全理解此选项的作用，你可能不需要它。

‘--restrict-file-names=modes’ ¶

更改在生成本地文件名时必须转义的远程URL中的字符。 此选项限制的字符将被转义，即替换为‘%HH’，其中‘HH’是对应于受限字符的十六进制数。 此选项也可用于强制所有字母大小写为小写或大写。

默认情况下，Wget会转义在您的操作系统上作为文件名部分无效或不安全的字符，以及通常不可打印的控制字符。 此选项对于更改这些默认设置很有用，可能是因为您正在下载到非本地分区，或者因为您想要禁用控制字符的转义，或者您想要进一步限制字符仅为ASCII值范围内的字符。

modes是一组以逗号分隔的文本值。 可接受的值是‘unix’、‘windows’、‘nocontrol’、‘ascii’、‘lowercase’和‘uppercase’。 值‘unix’和‘windows’是互斥的（一个会覆盖另一个），‘lowercase’和‘uppercase’也是互斥的。；。 最后两个是特殊情况，因为它们不会改变将被转义的字符集，而是强制本地文件路径转换为小写或大写。

当指定“unix”时，Wget会转义字符‘/’以及范围0–31和128–159内的控制字符。 这是在类Unix操作系统上的默认设置。

当给出“windows”时，Wget会转义字符‘\’、‘|’、‘/’、‘:’、‘?’、‘"’、‘*’、‘<’、‘>’以及范围0–31和128–159内的控制字符。 除此之外，Windows模式下的Wget使用‘+’而不是‘:’来分隔本地文件名中的主机和端口，并使用‘@’而不是‘?’来分隔文件名的查询部分与其余部分。 因此，在Unix模式下保存为‘www.xemacs.org:4300/search.pl?input=blah’的URL在Windows模式下将保存为‘www.xemacs.org+4300/search.pl@input=blah’。 此模式是Windows上的默认设置。

如果您指定‘nocontrol’，则控制字符的转义也会关闭。 当您下载包含UTF-8字符的URL名称时，此选项可能很有意义，在可以保存和显示UTF-8文件名的系统上（UTF-8字节序列中使用的一些可能字节值落在Wget指定的“控制”值范围内）。

‘ascii’模式用于指定任何值超出ASCII字符范围（即大于127）的字节应被转义。 这在保存编码与本地不匹配的文件名时可能很有用。

‘-4’ ¶
‘--inet4-only’；
‘-6’；
‘--inet6-only’；

强制连接到IPv4或IPv6地址。 使用‘--inet4-only’或‘-4’，Wget将仅连接到IPv4主机，忽略DNS中的AAAA记录，并拒绝连接到URL中指定的IPv6地址。 相反，使用‘--inet6-only’或‘-6’，Wget将仅连接到IPv6主机并忽略A记录和IPv4地址。

通常不需要这些选项。 默认情况下，支持IPv6的Wget将使用主机的DNS记录指定的地址族。 如果DNS响应包含IPv4和IPv6地址，Wget将依次尝试它们，直到找到一个可以连接的地址。 （另见下面描述的--prefer-family选项。）

这些选项可用于在双族系统上故意强制使用IPv4或IPv6地址族，通常用于辅助调试或处理损坏的网络配置。 ‘--inet6-only’和‘--inet4-only’不能同时指定。 在没有IPv6支持的Wget编译中，这两个选项都不可用。

‘--prefer-family=none/IPv4/IPv6’；

当有多个地址选择时，首先连接到指定地址族的地址。 默认情况下，使用DNS返回的地址顺序而不做任何更改。

这避免了从IPv4网络访问同时解析为IPv6和IPv4地址的主机时产生的虚假错误和连接尝试。 例如，‘www.kame.net’ 解析为 ‘2001:200:0:8002:203:47ff:fea5:3085’ 和 ‘203.178.141.194’。 当首选系列是 IPv4 时， IPv4 地址首先被使用；当首选系列是 IPv6 时， IPv6 地址首先被使用；如果指定的值是 none， 则使用 DNS 返回的地址顺序而不做更改。

与 ‘-4’ 和 ‘-6’ 不同，此选项不抑制访问 任何地址系列，它只改变访问地址的 顺序。 还要注意，此选项执行的重新排序是 稳定的—它不会影响相同系列的地址的顺序。 也就是说，所有 IPv4 地址 和所有 IPv6 地址的相对顺序在所有情况下都保持不变。

‘--retry-connrefused&rsquo；

将“连接被拒绝”视为暂时性错误并重试。 通常，当 Wget 无法连接到站点时，它会放弃 URL，因为连接失败被视为服务器根本没有运行的标志，重试不会有帮助。 此选项适用于镜像不可靠的站点，这些站点的服务器往往会短暂消失。

‘--user=user’ ¶
‘--password=password&rsquo；

为 FTP 和 HTTP 文件检索指定用户名 user 和密码 password。 这些参数可以被 ‘--ftp-user’ 和 ‘--ftp-password’ 选项覆盖用于 FTP 连接，以及 ‘--http-user’ 和 ‘--http-password’ 选项用于 HTTP 连接。

‘--ask-password&rsquo；

为每个建立的连接提示输入密码。 当使用 ‘--password’ 时不能指定，因为它们是互斥的。

‘--use-askpass=command&rsquo；

使用指定的命令提示输入用户和密码。 如果未指定命令，则使用环境变量 WGET_ASKPASS 中的命令。 如果未设置 WGET_ASKPASS，则使用环境变量 SSH_ASKPASS 中的命令。

您可以在 .wgetrc 中设置用于 use-askpass 的默认命令。 该 设置可以从命令行覆盖。

‘--no-iri’ ¶

关闭国际化 URI (IRI) 支持。 使用 ‘--iri’ 来 打开它。 IRI 支持默认是激活的。

您可以使用 .wgetrc 中的 iri 命令设置 IRI 支持的默认状态。 该设置可以从 命令行覆盖。

‘--local-encoding=encoding’ ¶

强制 Wget 使用 encoding 作为默认系统编码。 这影响 Wget 如何将作为参数指定的 URL 从区域设置转换为 UTF-8 以支持 IRI。

Wget 使用函数 nl_langinfo() 然后使用 CHARSET 环境变量来获取区域设置。 如果失败，则使用 ASCII。

您可以使用 .wgetrc 中的 local_encoding 命令设置默认的本地编码。 该设置可以从 命令行覆盖。

‘--remote-encoding=encoding’ ¶

强制 Wget 使用 encoding 作为默认远程服务器编码。 这影响 Wget 在递归获取期间如何将远程编码的 URI 转换为 UTF-8。 此选项仅对 IRI 支持有用，用于解释非 ASCII 字符。

对于 HTTP，远程编码可以在 HTTP Content-Type 头和 HTML Content-Type http-equiv 元标签中找到。

您可以使用 .wgetrc 中的 remoteencoding 命令设置默认编码。 该设置可以从命令行覆盖。

‘--unlink’ ¶

强制Wget取消链接文件而不是覆盖现有文件。 此选项对于下载到具有硬链接的目录非常有用。

2.6 目录选项 ¶

‘-nd’；
‘--no-directories’；

递归检索时不要创建目录层次结构。 启用此选项后，所有文件都将保存到当前目录，而不会覆盖（如果名称出现多次，文件名将获得扩展名‘.n’）。

‘-x’；
‘--force-directories’；

与‘-nd’相反——创建目录层次结构，即使原本不会创建。 例如。 ‘wget -x http://fly.srk.fer.hr/robots.txt’ 将下载的文件保存到 fly.srk.fer.hr/robots.txt。

‘-nH’；
‘--no-host-directories’；

禁用生成主机前缀的目录。 默认情况下，使用‘-r http://fly.srk.fer.hr/’调用Wget将创建以fly.srk.fer.hr/开头的目录结构。 此选项禁用此类行为。

‘--protocol-directories’；

使用协议名称作为本地文件名的目录组件。 例如，使用此选项，‘wget -r http://host’将保存到‘http/host/...’ 而不是仅仅到 ‘host/...’。

‘--cut-dirs=number’ ¶

忽略 number 目录组件。 这对于对递归检索将保存到的目录进行细粒度控制非常有用。

以‘ftp://ftp.xemacs.org/pub/xemacs/’目录为例；。 如果使用‘-r’检索它，它将被本地保存到ftp.xemacs.org/pub/xemacs/。 虽然‘-nH’选项可以移除ftp.xemacs.org/部分，但你仍然会卡在pub/xemacs。 这就是‘--cut-dirs’派上用场的地方；它使Wget“看不到”number远程目录组件。 以下是‘--cut-dirs’选项如何工作的几个例子。 

无选项        -> ftp.xemacs.org/pub/xemacs/
-nH               -> pub/xemacs/
-nH --cut-dirs=1  -> xemacs/
-nH --cut-dirs=2  -> 。

--cut-dirs=1      -> ftp.xemacs.org/xemacs/
……

如果你只是想摆脱目录结构，此选项类似于‘-nd’和‘-P’的组合；。 然而，与‘-nd’不同，‘--cut-dirs’不会丢失子目录——例如，使用‘-nH --cut-dirs=1’，一个beta/子目录将被放置到xemacs/beta，正如人们所期望的那样。

‘-P prefix’ ¶
‘--directory-prefix=prefix’；

将目录前缀设置为 prefix。 目录前缀是所有其他文件和子目录将保存到的目录，即检索树的顶部。 默认值为‘.’（当前目录）。

2.7 HTTP 选项 ¶

‘--default-page=name’ ¶

当文件名未知时（即，对于以斜杠结尾的URL），使用name作为默认文件名，而不是index.html。

‘-E’ ¶
‘--adjust-extension’

如果下载的文件类型为‘application/xhtml+xml’或‘text/html’，并且URL不以正则表达式‘\.[Hh][Tt][Mm][Ll]?’结尾，此选项将导致后缀‘.html’被附加到本地文件名。 例如，当您镜像一个使用‘.asp’页面的远程站点时，这很有用，但您希望镜像的页面可以在您的标准Apache服务器上查看。 另一个好的用途是当您下载CGI生成的材料时。 像‘http://site.com/article.cgi?25’这样的URL将被保存为article.cgi?25.html。

请注意，以这种方式更改的文件名每次重新镜像站点时都会重新下载，因为Wget无法判断本地X.html文件对应于远程URL ‘X’（因为它还不知道该URL生成的是‘text/html’或‘application/xhtml+xml’类型的输出。

从1.12版本开始，Wget还将确保任何下载的‘text/css’类型的文件以‘.css’后缀结尾，并且该选项从‘--html-extension’重命名，以更好地反映其新行为。 旧的选项名称仍然可以接受，但现在应该被视为已弃用。

从1.19.2版本开始，Wget还将确保任何下载的文件，如果其Content-Encoding为‘br’、‘compress’、‘deflate’或‘gzip’，则分别以‘.br’、‘.Z’、‘.zlib’和‘.gz’后缀结尾。

在未来的某个时候，此选项可能会扩展到包括其他类型内容的后缀，包括Wget不解析的内容类型。

‘--http-user=user’ ¶
‘--http-password=password’

在HTTP服务器上指定用户名user和密码password。 根据挑战的类型，Wget将使用basic（不安全）、digest或Windows NTLM认证方案对它们进行编码。

另一种指定用户名和密码的方法是在URL本身中（参见URL格式）。 无论哪种方法，都会将您的密码暴露给任何运行ps的人。 为了防止密码被看到，请使用‘--use-askpass’或将它们存储在.wgetrc或.netrc中，并确保使用chmod保护这些文件免受其他用户的访问。 如果密码非常重要，请不要将它们留在这些文件中——在Wget开始下载后编辑文件并删除它们。

‘--no-http-keep-alive’ ¶

关闭HTTP下载的“保持连接”功能。 通常，Wget会请求服务器保持连接打开，以便当您从同一服务器下载多个文档时，它们通过相同的TCP连接传输。 这节省了时间，同时减少了服务器的负载。

当由于某种原因，持久（保持连接）连接对您不起作用时，此选项很有用，例如由于服务器错误或服务器端脚本无法处理连接。

‘--no-cache’ ¶

禁用服务器端缓存。 在这种情况下，Wget将向远程服务器发送适当的指令（‘Cache-Control: no-cache’和‘Pragma: no-cache’）以从远程服务获取文件，而不是返回缓存版本。 这对于检索和刷新代理服务器上的过期文档特别有用。

默认允许缓存。

‘--no-cookies’ ¶

禁用cookies的使用。 Cookies是维护服务器端状态的机制。 服务器使用Set-Cookie头向客户端发送cookie，客户端在进一步的请求中使用相同的cookie进行响应。 由于cookies允许服务器所有者跟踪访问者，并且站点之间可以交换这些信息，一些人认为它们侵犯了隐私。 默认是使用cookies；但是，存储cookies默认是不开启的。

‘--load-cookies file’ ¶

在第一次HTTP检索之前从file加载cookies。 file是一个文本文件，格式最初由Netscape的cookies.txt文件使用。

当镜像需要登录才能访问部分或全部内容的站点时，通常会使用此选项。 登录过程通常通过Web服务器在接收并验证您的凭据后发出HTTP cookie来工作。 然后，当访问站点的该部分时，浏览器会重新发送cookie，从而证明您的身份。

镜像这样的站点需要Wget发送与您的浏览器在与站点通信时发送的相同cookies。 这是通过‘--load-cookies’实现的——只需将Wget指向cookies.txt文件的位置，它就会发送与您的浏览器在相同情况下发送的相同cookies。 不同的浏览器将文本cookie文件保存在不同的位置：

Netscape 4.x。

Cookies位于~/.netscape/cookies.txt。

Mozilla和Netscape 6.x。

Mozilla的cookie文件也命名为cookies.txt，位于~/.mozilla下的某个位置，在您的配置文件的目录中。 完整路径通常看起来像~/.mozilla/default/some-weird-string/cookies.txt。

Internet Explorer。

您可以通过文件菜单、导入和导出、导出Cookies来生成Wget可以使用的cookie文件。 这已在Internet Explorer 5中测试过；不能保证在早期版本中有效。

其他浏览器。

如果您使用其他浏览器创建cookies，‘--load-cookies’只有在您能够定位或生成Wget期望的Netscape格式的cookie文件时才会起作用。

如果您无法使用‘--load-cookies’，可能还有其他替代方案。 如果您的浏览器支持“cookie管理器”，您可以使用它来查看在镜像站点时使用的cookies。 记下cookie的名称和值，并手动指示Wget发送这些cookies，绕过“官方”cookie支持： 

wget --no-cookies --header "Cookie: name=value"
‘--save-cookies file’ ¶

在退出前将cookies保存到file。 这不会保存已过期或没有过期时间的cookies（所谓的“会话cookies”），但也可以参见‘--keep-session-cookies’。

‘--keep-session-cookies’ ¶

当指定时，会导致‘--save-cookies’也保存会话cookies。 会话cookies通常不会被保存，因为它们意味着保存在内存中并在退出浏览器时被遗忘。 保存它们对于需要登录或访问主页后才能访问某些页面的站点很有用。 使用此选项，多个Wget运行将被视为站点的单个浏览器会话。

由于cookie文件格式通常不携带会话cookies，Wget用过期时间戳0标记它们。 Wget的‘--load-cookies’会识别这些会话cookies，但它可能会混淆其他浏览器。 还要注意，这样加载的cookies将被视为其他会话cookies，这意味着如果您希望‘--save-cookies’再次保存它们，您必须再次使用‘--keep-session-cookies’。

‘--ignore-length’ ¶

不幸的是，一些HTTP服务器（更准确地说，是CGI程序）会发送虚假的Content-Length头，这会使Wget变得疯狂，因为它认为文档没有被完全检索。 如果Wget一次又一次地尝试获取相同的文档，每次都声称（否则正常的）连接在同一个字节上关闭，你就可以发现这种症状。

使用此选项，Wget将忽略Content-Length头——就像它从未存在过一样。

‘--header=header-line’ ¶

在每个HTTP请求中，与其余的头一起发送header-line。 提供的头按原样发送，这意味着它必须包含由冒号分隔的名称和值，并且不能包含换行符。

您可以通过多次指定‘--header’来定义多个额外的头。 

wget --header='Accept-Charset: iso-8859-2' \
     --header='Accept-Language: hr'        \
       http://fly.srk.fer.hr/

将空字符串指定为头的值将清除所有先前用户定义的头。

从Wget 1.10开始，此选项可用于覆盖自动生成的头。 此示例指示Wget连接到localhost，但在Host头中指定‘foo.bar’： 

wget --header="Host: foo.bar" http://localhost/

在Wget 1.10之前的版本中，这样使用‘--header’会导致发送重复的头。

‘--compression=type’ ¶

选择要使用的压缩类型。 合法值是‘auto’，‘gzip’和‘none’。

如果指定了‘auto’或‘gzip’，Wget会要求服务器使用gzip压缩格式压缩文件。 如果服务器压缩了文件并适当地设置了Content-Encoding头字段，文件将自动解压缩。

如果指定了‘none’，wget将不会要求服务器压缩文件，也不会解压缩任何服务器响应。 这是默认设置。

压缩支持目前是实验性的。 如果它被启用，请将任何错误报告给bug-wget@gnu.org。

‘--max-redirect=number’ ¶

指定资源的最大重定向次数。 默认值是20，这通常远远超过必要的次数。 然而，在那些您希望允许更多（或更少）重定向的场合，这是要使用的选项。

‘--proxy-user=user’ ¶
‘--proxy-password=password’

指定代理服务器上的用户名user和密码password进行认证。 Wget将使用basic认证方案对它们进行编码。

与‘--http-password’类似的安全考虑也适用于此。

‘--referer=url’ ¶

在HTTP请求中包含‘Referer: url’头。 对于检索那些假设它们总是被交互式网页浏览器检索并且只有在Referer设置为指向它们的页面之一时才能正确显示的文档非常有用。

‘--save-headers’ ¶

将HTTP服务器发送的头保存到文件中，在实际内容之前，以空行作为分隔符。

‘-U agent-string’ ¶
‘--user-agent=agent-string’

向HTTP服务器标识为agent-string。

HTTP协议允许客户端使用User-Agent头字段标识自己。 这使得可以区分 WWW软件，通常用于统计目的或跟踪 协议违规。 Wget通常标识为 ‘Wget/version’，version是Wget的当前版本 号。

然而，已知一些站点会根据User-Agent提供的信息定制输出。 虽然这在理论上不是一个坏主意，但它已被服务器滥用，拒绝向除（历史上）Netscape或更常见的Microsoft Internet Explorer之外的客户端提供信息。 此 选项允许您更改Wget发出的User-Agent行。 除非您真的知道自己在做什么，否则不建议使用此选项。

使用‘--user-agent=""’指定空用户代理指示Wget 不在HTTP请求中发送User-Agent头。

‘--post-data=string’ ¶
‘--post-file=file’；

使用POST作为所有HTTP请求的方法，并在请求体中发送指定的数据。 ‘--post-data’发送string作为数据，而‘--post-file’发送file的内容。 除此之外，它们的工作方式完全相同。 特别是， 它们都期望内容的形式为key1=value1&key2=value2， 特殊字符使用百分号编码；唯一的区别是 一个期望其内容作为命令行参数，另一个 从文件接受其内容。 特别是，‘--post-file’不 用于传输文件作为表单附件：这些必须 作为key=value数据（使用适当的百分号编码）出现，就像其他所有内容一样。 Wget目前不支持 multipart/form-data用于传输POST数据；仅支持 application/x-www-form-urlencoded。 只能指定 ‘--post-data’和‘--post-file’中的一个。

请注意，wget不要求内容的形式为 key1=value1&key2=value2，也不对其进行测试。 Wget将 简单地传输提供给它的任何数据。 然而，大多数服务器在处理HTML表单时期望 POST数据采用上述格式。

当使用‘--post-file’选项发送POST请求时，Wget将 文件视为二进制文件，并将在POST请求中发送每个字符， 而不去除尾随的换行符或换页符。 文本中的任何其他控制 字符也将按原样在POST请求中发送。

请注意，Wget需要提前知道POST数据的大小。 因此，--post-file的参数必须是常规 文件；指定FIFO或类似/dev/stdin的东西将不起作用。 目前尚不清楚如何绕过HTTP/1.0中固有的这一限制。 尽管HTTP/1.1引入了分块传输，不需要提前知道请求长度，但客户端不能 使用分块，除非它知道它正在与HTTP/1.1服务器通信。 而且它 在收到响应之前无法知道这一点，这又要求 请求已经完成——一个鸡生蛋蛋生鸡的问题。

注意：从1.15版本开始，如果Wget在POST请求完成后被重定向，其行为将取决于服务器返回的响应代码。 在301永久移动、302临时移动或 307临时重定向的情况下，Wget将根据RFC2616继续 发送POST请求。 如果服务器希望客户端在重定向时更改请求方法，它应该发送303 See Other响应代码。

此示例显示如何使用POST登录到服务器，然后继续下载 所需的页面，可能只有授权用户才能访问： 

# 登录到服务器。  这只能做一次。
wget --save-cookies cookies.txt \
     --post-data 'user=foo&password=bar' \
     http://example.com/auth.php

# 现在抓取我们关心的页面。
wget --load-cookies cookies.txt \
     -p http://example.com/interesting/article.php

如果服务器使用会话cookie来跟踪用户认证， 上述方法将不起作用，因为‘--save-cookies’不会保存 它们（浏览器也不会），cookies.txt文件将 为空。 在这种情况下，使用‘--keep-session-cookies’与 ‘--save-cookies’一起强制保存会话cookie。

‘--method=HTTP-Method’ ¶

为了RESTful脚本的目的，Wget允许发送其他HTTP方法， 而无需使用‘--header=Header-Line’显式设置它们；。 Wget将使用传递给‘--method’之后的任何字符串作为HTTP 方法发送到服务器。

‘--body-data=数据字符串’
‘--body-file=数据文件’

当需要向服务器发送额外数据时，必须设置此选项，与使用‘--method’指定的方法一起使用。 ‘--body-data’发送字符串作为数据，而‘--body-file’发送文件的内容。 除此之外，它们的工作方式完全相同。

目前，‘--body-file’不用于整体传输文件。 Wget目前不支持multipart/form-data传输数据；仅支持application/x-www-form-urlencoded。 未来可能会更改，以便wget将‘--body-file’作为完整文件发送，而不是将其内容发送到服务器。 请注意，Wget需要提前知道BODY数据的内容，因此‘--body-file’的参数应该是一个常规文件。 有关更详细的解释，请参见‘--post-file’。 只能指定‘--body-data’和‘--body-file’中的一个。

如果Wget在请求完成后被重定向，Wget将暂停当前方法并发送GET请求，直到重定向完成。 这对于所有重定向响应代码都是正确的，除了307临时重定向，它用于明确指定请求方法不应更改。 另一个例外是当方法设置为POST时，在这种情况下，遵循‘--post-data’下指定的重定向规则。

‘--content-disposition’ ¶

如果设置为on，则启用对Content-Disposition头的实验性（非全功能）支持。 这目前可能导致HEAD请求的额外往返服务器，并且已知存在一些错误，这就是为什么它目前默认未启用的原因。

此选项对于某些使用Content-Disposition头来描述下载文件应具有的名称的文件下载CGI程序很有用。

当与‘--metalink-over-http’和‘--trust-server-names’结合使用时，如果可用，‘Content-Type: application/metalink4+xml’文件使用Content-Disposition文件名字段命名。

‘--content-on-error’ ¶

如果设置为on，当服务器响应表示错误的http状态代码时，wget不会跳过内容。

‘--trust-server-names’ ¶

如果设置此选项，重定向时，本地文件名将基于重定向URL。 默认情况下，本地文件名基于原始URL。 在进行递归检索时，这可能会有帮助，因为在许多网站中，重定向的URL对应于底层文件结构，而链接URL则不然。

‘--auth-no-challenge’ ¶

如果给出此选项，Wget将为所有请求发送基本HTTP认证信息（明文用户名和密码），就像Wget 1.10.2及更早版本默认所做的那样。

不建议使用此选项，仅用于支持一些少数晦涩的服务器，这些服务器从不发送HTTP认证挑战，但接受未经请求的认证信息，例如，除了基于表单的认证之外。

‘--retry-on-host-error’

将主机错误，如“名称解析中的临时失败”，视为非致命、暂时性错误。

‘--retry-on-http-error=code[,code,...]’

将给定的HTTP响应代码视为非致命、暂时性错误。 提供一个以逗号分隔的3位HTTP响应代码列表作为参数。 用于处理需要重试但服务器响应通常不被Wget重试的错误代码的特殊情况。 此类错误可能是503（服务不可用）和429（请求过多）。 此选项启用的重试受Wget的正常重试时间和重试次数限制。

使用此选项旨在支持特殊用例，通常不推荐使用，因为它可以强制重试，即使服务器实际上试图减少其负载。 请明智使用，并且仅在您知道自己在做什么时使用。

2.8 HTTPS (SSL/TLS) 选项 ¶

为了支持加密的HTTP（HTTPS）下载，Wget必须使用外部SSL库编译。 当前的默认值是GnuTLS。 此外，Wget还支持HSTS（HTTP严格传输安全）。 如果Wget在没有SSL支持的情况下编译，这些选项都不可用。

‘--secure-protocol=protocol’ ¶

选择要使用的安全协议。 合法值为‘auto’、‘SSLv2’、‘SSLv3’、‘TLSv1’、‘TLSv1_1’、‘TLSv1_2’、‘TLSv1_3’和‘PFS’。 如果使用‘auto’，SSL库将自动选择适当的协议，这是通过发送TLSv1问候实现的。 这是默认设置。

指定‘SSLv2’、‘SSLv3’、‘TLSv1’、‘TLSv1_1’、‘TLSv1_2’或‘TLSv1_3’将强制使用相应的协议。 这在处理旧的、有bug的SSL服务器实现时非常有用，这些实现使得底层的SSL库难以选择正确的协议版本。 幸运的是，这样的服务器相当罕见。

指定‘PFS’将强制使用所谓的完美前向安全密码套件。 简而言之，PFS通过为每个SSL连接创建一次性密钥来增加安全性。 它对客户端和服务器有更多的CPU影响。 我们使用已知的安全密码（例如，没有MD4）和TLS协议。 此模式还明确排除了非PFS密钥交换方法，如RSA。

‘--https-only’；

在递归模式下，只跟随HTTPS链接。

‘--ciphers’；

设置密码列表字符串。 通常，此字符串设置用户希望使用的密码套件和其他SSL/TLS选项，按照优先顺序（GnuTLS称之为“优先级字符串”）。 此字符串将原封不动地传递给SSL/TLS引擎（OpenSSL或GnuTLS），因此其格式和语法取决于该引擎。 Wget不会以任何方式处理或操作它。 有关更多信息，请参阅OpenSSL或GnuTLS文档。

‘--no-check-certificate’ ¶

不要根据可用的证书颁发机构检查服务器证书。 也不要要求URL主机名与证书提供的通用名匹配。

自Wget 1.10起，默认行为是根据认可的证书颁发机构验证服务器的证书，如果验证失败，则中断SSL握手并中止下载。 虽然这提供了更安全的下载，但它确实破坏了与一些使用自签名、过期或其他无效证书的站点的互操作性，这些站点在以前的Wget版本中工作。 此选项强制进入“不安全”操作模式，将证书验证错误转换为警告并允许您继续。

如果您遇到“证书验证”错误或“通用名与请求的主机名不匹配”的错误，您可以使用此选项绕过验证并继续下载。 只有在您确信站点的真实性，或者您真的不关心其证书的有效性时，才使用此选项。 在传输机密或重要数据时，不检查证书几乎总是一个坏主意。 对于自签名/内部证书，您应该下载证书并对其进行验证，而不是强制使用此不安全模式。 如果您确实不希望进行任何证书验证，可以指定–check-certificate=quiet来告诉wget不要打印任何关于无效证书的警告，尽管在大多数情况下这样做是错误的。

‘--certificate=file’ ¶

使用存储在file中的客户端证书。 这对于配置为要求连接客户端提供证书的服务器是必需的。 通常不需要证书，此开关是可选的。

‘--certificate-type=type’ ¶

指定客户端证书的类型。 合法值为‘PEM’（默认假设）和‘DER’，也称为‘ASN1’。

‘--private-key=file’；

从file读取私钥。 这允许您在证书之外的文件中提供私钥。

‘--private-key-type=type’；

指定私钥的类型。 接受的值为‘PEM’（默认）和‘DER’。

‘--ca-certificate=file’；

使用file作为包含证书颁发机构（“CA”）捆绑包的文件以验证对等方。 证书必须为PEM格式。

没有此选项，Wget会在OpenSSL安装时选择的系统指定位置查找CA证书。

‘--ca-directory=directory’ ¶

指定包含PEM格式CA证书的目录。 每个文件包含一个CA证书，文件名基于从证书派生的哈希值。 这是通过使用OpenSSL提供的c_rehash实用程序处理证书目录实现的。 当安装了许多证书时，使用‘--ca-directory’比‘--ca-certificate’更有效，因为它允许Wget按需获取证书。

没有此选项，Wget会在OpenSSL安装时选择的系统指定位置查找CA证书。

‘--crl-file=file’ ¶

在file中指定CRL文件。 这对于已被CA撤销的证书是必需的。

‘--pinnedpubkey=file/hashes’ ¶

告诉wget使用指定的公钥文件（或哈希）来验证对等方。 这可以是包含PEM或DER格式的单个公钥的文件路径，或者是任何数量的base64编码的sha256哈希，前面加上“sha256//”并用“;”分隔；

当协商TLS或SSL连接时，服务器发送一个证书表明其身份。 从该证书中提取公钥，如果它与提供给此选项的公钥不完全匹配，wget将在发送或接收任何数据之前中止连接。

‘--random-file=file’ ¶

[仅限OpenSSL和LibreSSL] 使用file作为随机数据的源，用于在没有/dev/urandom的系统上种子伪随机数生成器。

在这样的系统上，SSL库需要一个外部随机源来初始化。 随机性可以由EGD提供（见下面的‘--egd-file’）或从用户指定的外部源读取。 如果未指定此选项，Wget 将在 $RANDFILE 中查找随机数据，如果未设置，则在 $HOME/.rnd 中查找。

如果您遇到“无法为 OpenSSL PRNG 提供种子；禁用 SSL。”错误，您应该使用上述方法之一提供随机数据。

‘--egd-file=file’ ¶

[仅限 OpenSSL] 使用 file 作为 EGD 套接字。 EGD 代表 熵收集守护进程，一个用户空间程序，它从各种不可预测的系统源收集数据，并使其可供可能需要它的其他程序使用。 加密软件，如 SSL 库，需要非重复随机性的来源来为用于生成加密强密钥的随机数生成器提供种子。

OpenSSL 允许用户使用 RAND_FILE 环境变量指定自己的熵源。 如果未设置此变量，或者指定的文件未产生足够的随机性，OpenSSL 将从使用此选项指定的 EGD 套接字读取随机数据。

如果未指定此选项（且未使用等效的启动命令），则永远不会联系 EGD。 在现代支持 /dev/urandom 的 Unix 系统上不需要 EGD。

‘--no-hsts’ ¶

Wget 默认支持 HSTS（HTTP 严格传输安全，RFC 6797）。 使用 ‘--no-hsts’ 使 Wget 作为不符合 HSTS 的用户代理。 因此，Wget 将忽略所有 Strict-Transport-Security 头，并且不会强制执行任何现有的 HSTS 策略。

‘--hsts-file=file’

默认情况下，Wget 将其 HSTS 数据库存储在 ~/.wget-hsts 中。 您可以使用 ‘--hsts-file’ 来覆盖此设置。 Wget 将使用提供的文件作为 HSTS 数据库。 此类文件必须符合 Wget 使用的正确 HSTS 数据库格式。 如果 Wget 无法解析提供的文件，则行为未指定。

Wget 的 HSTS 数据库是一个纯文本文件。 每行包含一个 HSTS 条目（即已发出 Strict-Transport-Security 头并因此指定了要应用的具体 HSTS 策略的站点）。 以破折号（#）开头的行将被 Wget 忽略。 请注意，尽管这种方便的人类可读性手动修改 HSTS 数据库通常不是一个好主意。

HSTS 条目行由几个由空格分隔的字段组成：

<hostname> SP [<port>] SP <include subdomains> SP <created> SP <max-age>

hostname 和 port 字段指示给定 HSTS 策略适用的主机名和端口。 port 字段可能为零，并且在大多数情况下都是如此。 这意味着在决定是否应在给定请求上应用此类 HSTS 策略时，不会考虑端口号（仅评估主机名）。 当 port 不为零时，将评估目标主机名和端口，并且仅当两者都匹配时才会应用 HSTS 策略。 此功能仅用于测试/开发目的。 Wget 测试套件（在 testenv/ 中）创建具有显式端口的 HSTS 数据库，以确保 Wget 的正确行为。 RFC 6797 不鼓励将 HSTS 策略应用于非默认端口（参见附录 B“HSTS 策略与同源策略之间的差异”）。 因此，此功能不应在生产环境中使用，port 通常为零。 最后三个字段执行它们预期的操作。 字段 include_subdomains 可以是 1 或 0，它指示目标域的子域是否也应包含在给定的 HSTS 策略中。 created 和 max-age 字段 保存了该条目创建时（首次被Wget看到时）的时间戳值和HSTS定义的’max-age’值，该值说明了该HSTS策略应保持活动的时间长度， 以自created中存储的时间戳以来经过的秒数来衡量。 一旦该时间 过去，该HSTS策略将不再有效，并最终从数据库中删除。

如果您通过‘--hsts-file’提供自己的HSTS数据库，请注意，如果远程服务器请求的HSTS策略与文件中的策略之间发生任何变化，Wget 可能会修改提供的文件。 当Wget退出时， 它通过用新条目重写数据库文件来有效地更新HSTS数据库。

如果提供的文件不存在，Wget将创建一个。 该文件将包含新的HSTS 条目。 如果没有生成HSTS条目（没有任何服务器发送Strict-Transport-Security头）， 则不会创建文件，甚至不会创建空文件。 这种行为也适用于默认的数据库文件（~/.wget-hsts）：直到某个服务器强制执行HSTS策略，它才会被创建。

注意不要覆盖可能由其他Wget进程同时进行的HSTS数据库更改。 在将更新的HSTS条目转储到文件之前，Wget将重新读取它并合并更改。

不鼓励使用自定义HSTS数据库和/或修改现有数据库。 有关此类做法可能带来的安全威胁的更多信息， 请参阅RFC 6797的第14节“安全考虑”，特别是第14.9节“HSTS策略存储的创造性操作”；。

‘--warc-file=file’；

使用file作为目标WARC文件。

‘--warc-header=string’；

使用string作为warcinfo记录。

‘--warc-max-size=size’；

将WARC文件的最大大小设置为size。

‘--warc-cdx’；

写入CDX索引文件。

‘--warc-dedup=file’；

不要存储此CDX文件中列出的记录。

‘--no-warc-compression’；

不要用GZIP压缩WARC文件。

‘--no-warc-digests’；

不计算SHA1摘要。

‘--no-warc-keep-log’；

不要将日志文件存储在WARC记录中。

‘--warc-tempdir=dir’；

指定WARC写入器创建的临时文件的位置。

2.9 FTP选项 ¶

‘--ftp-user=user’ ¶
‘--ftp-password=password’；

在FTP服务器上指定用户名user和密码password。 如果没有这个，或者相应的启动选项， 密码默认为‘-wget@’，通常用于匿名 FTP。

另一种指定用户名和密码的方法是在URL本身中 （参见URL格式）。 无论哪种方法都会向任何运行ps的人泄露你的密码。 为了防止密码被看到，将它们存储在.wgetrc或.netrc中，并确保使用chmod保护这些文件免受其他用户的访问。 如果密码非常重要，也不要将它们留在这些文件中——编辑文件并在Wget开始下载后删除它们。

‘--no-remove-listing’ ¶

不要删除由FTP检索生成的临时.listing文件。 通常，这些文件包含从FTP服务器接收的原始目录列表。 不删除它们对于调试目的或当您希望轻松检查远程服务器目录内容时（例如，验证您运行的镜像是否完整）非常有用。

请注意，即使Wget为此文件写入已知文件名，在用户将.listing符号链接到/etc/passwd或类似文件并要求root在其目录中运行Wget的情况下，这也不是安全漏洞。 根据使用的选项，Wget将拒绝写入.listing，使通配/递归/时间戳操作失败，或者符号链接将被删除并替换为实际的.listing文件，或者列表将被写入.listing.number文件。

即使这种情况不是问题，root也不应在不受信任的用户目录中运行Wget。 用户可以简单地将index.html链接到/etc/passwd并要求root使用‘-N’或‘-r’运行Wget，以便文件被覆盖。

‘--no-glob’ ¶

关闭FTP通配。 通配指的是使用类似shell的特殊字符（通配符），如‘*’、‘?’、‘[’ 和‘]’来一次性从同一目录检索多个文件，如： 

wget ftp://gnjilux.srk.fer.hr/*.msg

默认情况下，如果URL包含通配字符，通配将被打开。 此选项可用于永久打开或关闭通配。

您可能需要引用URL以保护它不被您的shell扩展。 通配使Wget寻找目录列表，这是系统特定的。 这就是为什么它目前仅适用于Unix FTP 服务器（以及那些模拟Unix ls输出的服务器）。

‘--no-passive-ftp’ ¶

禁用被动FTP传输模式的使用。 被动FTP 要求客户端连接到服务器以建立数据连接，而不是相反。

如果机器直接连接到互联网，被动和主动FTP应该同样有效。 在大多数防火墙和NAT 配置后面，被动FTP有更好的工作机会。 然而， 在一些罕见的防火墙配置中，当被动FTP不工作时，主动FTP实际上可以工作。 如果您怀疑是这种情况，请使用此 选项，或在您的初始化文件中设置passive_ftp=off。

‘--preserve-permissions’ ¶

保留远程文件权限而不是由umask设置的权限。

‘--retr-symlinks’ ¶

默认情况下，当递归检索FTP目录并遇到符号链接时，符号链接被遍历并检索指向的文件。 目前，Wget不会遍历符号链接到目录以递归下载它们，尽管此功能可能会在未来添加。

当指定‘--retr-symlinks=no’时，不会下载链接到的文件。 相反，本地文件系统上会创建一个匹配的符号链接。 除非此递归检索会单独遇到并下载它，否则不会检索指向的文件。 此 选项存在安全风险，恶意FTP服务器可能通过特制的 .LISTING文件导致Wget写入到预期目录之外的文件。

请注意，当因为文件在命令行上指定而不是因为递归到而检索文件时， 此选项无效。 在这种情况下，符号链接总是被遍历。

2.10 FTPS 选项 ¶

‘--ftps-implicit’

此选项告诉Wget隐式使用FTPS。 隐式FTPS包括从控制连接的一开始就初始化SSL/TLS。 此选项不会发送AUTH TLS命令：它假设服务器支持FTPS并直接开始SSL/TLS连接。 如果尝试成功，会话将继续，就像常规的FTPS一样（发送PBSZ和PROT等）。 隐式FTPS不再是FTPS实现的要求，因此许多服务器可能不支持它。 如果传递了‘--ftps-implicit’并且没有指定明确的端口号，将使用隐式FTPS的默认端口990，而不是“正常”（显式）FTPS的默认端口，后者与FTP相同，为21。

‘--no-ftps-resume-ssl’

不在数据通道中恢复SSL/TLS会话。 当开始数据连接时，Wget尝试恢复之前在控制连接中开始的SSL/TLS会话。 SSL/TLS会话恢复通过重用前一个会话的SSL/TLS参数来避免执行全新的握手。 通常，FTPS服务器希望如此，因此Wget默认这样做。 然而，在极少数情况下，可能希望在每次数据连接中开始一个全新的SSL/TLS会话。 这就是‘--no-ftps-resume-ssl’的用途。

‘--ftps-clear-data-connection’

所有数据连接将以纯文本形式进行。 只有控制连接将在SSL/TLS下进行。 Wget将发送PROT C命令来实现这一点，这必须得到服务器的批准。

‘--ftps-fallback-to-ftp’

如果目标服务器不支持FTPS，则回退到FTP。 出于安全原因，默认情况下不启用此选项。 默认行为是退出并报错。 如果服务器没有成功回复初始的AUTH TLS命令，或者在隐式FTPS的情况下，如果初始SSL/TLS连接尝试被拒绝，则认为该服务器不支持FTPS。

2.11 递归检索选项 ¶

‘-r’
‘--recursive’

开启递归检索。 参见递归下载，了解更多详情。 默认的最大深度为5。

‘-l depth’
‘--level=depth’

将Wget将递归进入的子目录的最大数量设置为depth。 为了防止在使用递归时意外下载非常大的网站，默认情况下限制为深度5，即从提供的URL开始，最多遍历5层目录。 设置‘-l 0’或‘-l inf’以进行无限递归深度。 

wget -r -l 0 http://site/1.html

理想情况下，人们会期望这只会下载1.html。 但不幸的是，情况并非如此，因为‘-l 0’等同于‘-l inf’—即无限递归。 要下载单个HTML页面（或少量页面），请在命令行上指定所有页面，并省略‘-r’和‘-l’。 要下载查看单个HTML页面所需的基本项目，请参见‘页面必需品’。

‘--delete-after’ ¶

此选项告诉Wget在下载每个文件后删除它，之后。 通过代理预取热门页面非常有用，例如： 

wget -r -nd --delete-after http://whatever.com/~popular/page/

‘-r’ 选项用于递归检索，‘-nd’ 用于不创建目录。

请注意，‘--delete-after’ 会删除本地机器上的文件。 例如，它不会向远程FTP站点发出‘DELE’命令。 还要注意的是，当指定了‘--delete-after’时，‘--convert-links’会被忽略，因此‘.orig’文件根本不会被创建。

‘-k’ ¶
‘--convert-links’；

下载完成后，转换文档中的链接，使其适合本地查看。 这不仅影响可见的超链接，还影响文档中链接到外部内容的任何部分，如嵌入的图像、样式表的链接、非HTML内容的超链接等。

每个链接将以以下两种方式之一进行更改：

  • Wget下载的文件链接将更改为指向它们所指向文件的相对链接。

    例如：如果下载的文件/foo/doc.html链接到/bar/img.gif，也被下载，那么doc.html中的链接将被修改为指向‘../bar/img.gif’。 这种转换对于任意目录组合都能可靠地工作。

  • Wget未下载的文件链接将更改为包含它们指向位置的主机名和绝对路径。

    例如：如果下载的文件/foo/doc.html链接到/bar/img.gif（或链接到../bar/img.gif），那么doc.html中的链接将被修改为指向http://hostname/bar/img.gif。

因此，本地浏览工作可靠：如果链接的文件已下载，链接将引用其本地名称；如果未下载，链接将引用其完整的互联网地址，而不是呈现一个断开的链接。 前者的链接被转换为相对链接，确保您可以将下载的层次结构移动到另一个目录。

请注意，只有在下载结束时，Wget才能知道哪些链接已被下载。 因此，‘-k’所做的工作将在所有下载结束时执行。

‘--convert-file-only’；

此选项仅转换URL的文件名部分，其余部分保持不变。 这个文件名部分有时被称为“basename”，尽管我们在这里避免使用这个术语以免引起混淆。

它与‘--adjust-extension’配合使用效果特别好，尽管这种配合不是强制的。 它对于用从不同主机下载的文件填充互联网缓存非常有用。

例如：如果某个链接指向//foo.com/bar.cgi?xyz，并且断言了‘--adjust-extension’，其本地目的地打算是./foo.com/bar.cgi?xyz.css，那么链接将被转换为//foo.com/bar.cgi?xyz.css。 请注意，只有文件名部分被修改。 URL的其余部分保持不变，包括网络路径（//），否则将由Wget处理并转换为有效方案（即http://）。

‘-K’ ¶
‘--backup-converted’；

转换文件时，使用‘.orig’后缀备份原始版本。 影响‘-N’的行为（参见HTTP时间戳内部）。

‘-m’；
‘--mirror’；

开启适合镜像的选项。 此选项开启递归和时间戳，设置无限递归深度并保留FTP目录列表。 它目前等同于‘-r -N -l inf --no-remove-listing’。

‘-p’ ¶
‘--page-requisites’

此选项使Wget下载所有必要的文件以正确显示给定的HTML页面。 这包括内联图像、声音和引用的样式表等内容。

通常，当下载单个HTML页面时，任何可能需要正确显示它的必要文档都不会被下载。 使用 ‘-r’与‘-l’一起可以帮助，但由于Wget通常不区分外部和内联文档，通常会留下缺少其必要条件的“叶子文档”。

例如，假设文档1.html包含一个引用1.gif的<IMG>标签 和一个指向外部文档2.html的<A>标签。 假设2.html类似，但其图像是2.gif，并且它链接到3.html。 假设这种情况继续到某个任意高的数字。

如果执行命令： 

wget -r -l 2 http://site/1.html

那么1.html、1.gif、2.html、2.gif和 3.html将被下载。 如您所见，3.html缺少其必要条件3.gif，因为Wget只是计算从1.html开始的跳数（最多2次）以确定停止递归的位置。 但是，使用此命令： 

wget -r -l 2 -p http://site/1.html

所有上述文件和3.html的必要条件3.gif 将被下载。 同样， 

wget -r -l 1 -p http://site/1.html

将导致1.html、1.gif、2.html和2.gif 被下载。 有人可能会认为： 

wget -r -l 0 -p http://site/1.html

只会下载1.html和1.gif，但不幸的是 情况并非如此，因为‘-l 0’等同于 ‘-l inf’—即无限递归。 要下载单个HTML 页面（或在命令行或‘-i’URL输入文件中指定的少数页面）及其必要条件，只需省略 ‘-r’和‘-l’： 

wget -p http://site/1.html

请注意，Wget将表现得好像指定了‘-r’，但只会下载该单个页面及其必要条件。 从该页面到外部文档的链接将不会被跟随。 实际上，要下载 单个页面及其所有必要条件（即使它们存在于不同的网站上），并确保本地正确显示，作者 喜欢使用一些额外的选项： 

wget -E -H -k -K -p http://site/document

要结束这个话题，值得知道的是，Wget认为的外部文档链接是任何在<A>标签、 <AREA>标签或<LINK>标签中指定的URL，除了<LINK REL="stylesheet">。

‘--strict-comments’ ¶

开启对HTML评论的严格解析。 默认是在第一次出现‘-->’时终止评论。

根据规范，HTML评论被表达为SGML 声明。 声明是以 ‘<!’开始并以‘>’结束的特殊标记，如‘<!DOCTYPE ...>’， 可能包含在一对‘--’分隔符之间的评论。 HTML 评论是“空声明”，即没有任何非评论文本的SGML声明。 因此，‘<!--foo-->’是一个有效的评论， ‘<!--one-- --two-->’也是，但‘<!--1--2-->’不是。

另一方面，大多数HTML编写者并不将注释视为除了用‘<!--’和‘-->’分隔的文本之外的任何东西，这并不完全相同。 例如，像‘<!------------>’这样的东西，只要破折号的数量是四的倍数（！），就可以作为有效的注释。 如果不是，注释技术上将持续到下一个‘--’，这可能位于文档的另一端。 因此，许多流行的浏览器完全忽略了规范，并实现了用户所期望的：用‘<!--’和‘-->’分隔的注释；。

直到1.9版本，Wget严格解释注释，这导致许多网页中缺少链接，这些网页在浏览器中显示良好，但不幸包含了不符合规范的注释。 从1.9版本开始，Wget加入了实现“天真”注释的客户端行列，每次注释在第一次出现‘-->’时终止；。

如果出于某种原因，您想要严格的注释解析，请使用此选项来开启它。

3 递归下载 ¶

GNU Wget能够遍历Web的部分（或单个HTTP或FTP服务器），跟随链接和目录结构。 我们将其称为递归检索，或递归。

使用HTTP URLs，Wget从给定的URL检索并解析HTML或CSS，通过标记如href或src，或使用‘url()’函数符号指定的CSS URI值检索文档引用的文件。 如果新下载的文件也是text/html，application/xhtml+xml，或text/css类型，它将被解析并进一步跟随。

HTTP和HTML/CSS内容的递归检索是广度优先的。 这意味着Wget首先下载请求的文档，然后下载该文档链接的文档，然后下载它们链接的文档，依此类推。 换句话说，Wget首先下载深度为1的文档，然后是深度为2的文档，依此类推，直到指定的最大深度。

检索可能下降的最大深度由‘-l’选项指定。 默认的最大深度是五层。

当递归检索FTP URL时，Wget将从远程服务器上的给定目录树（包括子目录，直到指定的深度）检索所有数据，在本地创建其镜像。 FTP检索也受到depth参数的限制。 与HTTP递归不同，FTP递归是深度优先的。

默认情况下，Wget将创建一个本地目录树，与远程服务器上找到的目录树相对应。

递归检索可以找到许多应用，其中最重要的是镜像。 它对于WWW演示以及任何其他通过本地存储文件绕过慢速网络连接的机会也很有用。

您应该被警告，递归下载可能会使远程服务器过载。 因此，许多管理员对此表示不满，如果他们检测到从您的站点快速下载大量内容，可能会禁止访问。 从互联网服务器下载时，请考虑使用‘-w’选项在访问服务器之间引入延迟。 下载将花费更长的时间，但服务器管理员不会因为您的粗鲁行为而感到惊慌。

当然，递归下载可能会在您的机器上引起问题。 如果任其运行，它很容易填满磁盘。 如果从本地网络下载，它还会占用系统带宽，以及消耗内存和CPU。

尝试指定与您尝试实现的下载类型匹配的标准。 如果您只想下载一个页面，请使用‘--page-requisites’而不进行任何额外的递归。 如果您想下载一个目录下的内容，请使用‘-np’以避免下载其他目录的内容。 如果您想从一个目录下载所有文件，请使用‘-l 1’以确保递归深度不超过一。 参见跟随链接，了解更多信息。

递归检索应谨慎使用。 不要说您没有被警告。

4 跟随链接 ¶

当递归检索时，不希望检索大量不必要的数据。 大多数时候，用户确切地记得他们想要下载的内容，并希望Wget只跟随特定的链接。

例如，如果您希望从‘fly.srk.fer.hr’下载音乐存档，您不会希望下载所有主页，这些主页恰好被存档的某个不显眼部分引用。

  • 跨越主机
  • 文件类型
  • 基于目录的限制
  • 相对链接
  • 跟随FTP链接

4.1 跨越主机 ¶

Wget的递归检索通常拒绝访问与命令行指定的主机不同的主机。 这是一个合理的默认设置；没有它，每次检索都有可能将您的Wget变成一个小型的谷歌。

然而，访问不同的主机，或主机跨越，有时是一个有用的选项。 也许图像是从不同的服务器提供的。 也许您正在镜像一个由三个服务器之间相互链接的页面组成的站点。 也许服务器有两个等效的名称，而HTML页面交替引用两者。

跨越到任何主机—‘-H’；

‘-H’选项开启主机跨越，从而允许Wget的递归运行访问任何链接引用的主机。 除非应用了足够的递归限制标准深度，否则这些外部主机通常会链接到更多的主机，依此类推，直到Wget最终吸收比您预期更多的数据。

限制跨越到某些域—‘-D’；

‘-D’选项允许您指定将被跟随的域，从而将递归限制仅属于这些域的主机。 显然，这仅在结合‘-H’时才有意义；。 一个典型的例子是下载‘www.example.com’的内容，但允许从‘images.example.com’等下载： 

wget -rH -Dexample.com http://www.example.com/

您可以通过用逗号分隔来指定多个地址，例如‘-Ddomain1.com,domain2.com’。

保持下载远离某些域—‘--exclude-domains’；

如果有您想特别排除的域，您可以使用‘--exclude-domains’，它接受与‘-D’相同类型的参数，但将排除所有列出的域。 例如，如果您想从‘foo.edu’域下载所有主机，除了‘sunsite.foo.edu’，您可以这样做： 

wget -rH -Dfoo.edu --exclude-domains sunsite.foo.edu \
    http://www.foo.edu/

4.2 文件类型 ¶

从网络下载材料时，您通常希望将检索限制为仅某些文件类型。 例如，如果您对下载GIFs感兴趣，您不会高兴得到大量的PostScript文档，反之亦然。

Wget提供了两个选项来处理这个问题。 每个选项描述列出了短名称、长名称和.wgetrc中的等效命令。

‘-A acclist’；
‘--accept acclist’；
‘accept = acclist’；
‘--accept-regex urlregex’；
‘accept-regex = urlregex’；

‘--accept’选项的参数是Wget在递归检索期间将下载的文件后缀或模式的列表。 后缀是文件的结尾部分，由“正常”字母组成，例如‘gif’或‘.jpg’。 匹配模式包含类似shell的通配符，例如‘books*’或‘zelazny*196[0-9]*’。

因此，指定‘wget -A gif,jpg’将使Wget仅下载以‘gif’或‘jpg’结尾的文件，即GIF和JPEG。 另一方面，‘wget -A "zelazny*196[0-9]*"’将仅下载以‘zelazny’开头并在任何位置包含1960到1969数字的文件。 查看您的shell手册以了解模式匹配的工作原理。

当然，任何数量的后缀和模式都可以组合成逗号分隔的列表，并作为‘-A’的参数给出；。

‘--accept-regex’选项的参数是与完整URL匹配的正则表达式。

‘-R rejlist’ ¶
‘--reject rejlist’；
‘reject = rejlist’；
‘--reject-regex urlregex’；
‘reject-regex = urlregex’；

‘--reject’选项的工作方式与‘--accept’相同，只是其逻辑相反；Wget将下载所有文件除了与列表中的后缀（或模式）匹配的文件。

因此，如果您想下载整个页面除了繁琐的MPEG和.AU文件，您可以使用‘wget -R mpg,mpeg,au’。 类似地，要下载所有文件除了以‘bjork’开头的文件，使用‘wget -R "bjork*"’。 引号用于防止shell扩展。

‘--accept-regex’选项的参数是与完整URL匹配的正则表达式。

‘-A’和‘-R’选项可以组合使用，以实现更好的文件检索微调。 例如。 ‘wget -A "*zelazny*" -R .ps’将下载所有名称中包含‘zelazny’的文件，但不包括PostScript文件。

请注意，这两个选项不会影响HTML文件的下载（由‘.htm’或‘.html’文件名前缀确定）。 这种行为可能不适合所有用户，并且可能会在Wget的未来版本中更改。

请注意，查询字符串（URL末尾以问号开头的字符串）不包括在用于接受/拒绝规则的文件名中，尽管这些实际上会影响到本地文件的名称选择。 预计Wget的未来版本将提供一个选项，允许匹配查询字符串。

最后，值得注意的是，接受/拒绝列表会与下载的文件进行两次匹配：一次与URL的文件名部分匹配，以确定是否应首先下载该文件；然后，在它被接受并成功下载后，本地文件的名称也会与接受/拒绝列表进行检查，以查看是否应删除。 理由是，由于‘.htm’和‘.html’文件总是被下载，无论接受/拒绝规则如何，如果它们确实匹配接受/拒绝列表，它们应该在下载并扫描链接后被删除。 然而，这可能会导致意外结果，因为本地文件名可能与原始URL文件名在以下方面不同，所有这些都可能改变接受/拒绝规则是否匹配：

  • 如果本地文件已存在且指定了‘--no-directories’，则会在原始名称后附加数字后缀。
  • 如果指定了‘--adjust-extension’，则本地文件名可能会附加‘.html’。 如果Wget以‘-E -A.php’调用，则文件名如‘index.php’将被接受，但在下载时将命名为‘index.php.html’，这不再匹配，因此文件将被删除。
  • 查询字符串不参与URL匹配，但包含在本地文件名中，因此确实参与文件名匹配。

这种行为也被认为不太理想，并且可能会在Wget的未来版本中更改。

4.3 基于目录的限制 ¶

不管其他链接跟随设施如何，基于文件所在的目录来限制检索哪些文件通常很有用。 这样做可能有很多原因——主页可能组织在一个合理的目录结构中；或者某些目录可能包含无用的信息，例如 /cgi-bin 或 /dev 目录。

Wget提供了三种不同的选项来处理这个需求。 每个选项描述都列出了短名称、长名称和 .wgetrc 中的等效命令。

‘-I list’
‘--include list’
‘include_directories = list’

‘-I’ 选项接受一个逗号分隔的目录列表，这些目录包含在检索中。 任何其他目录将被简单地忽略。 目录是绝对路径。

所以，如果你想从 ‘http://host/people/bozo/’ 下载，只跟随指向 /people 目录中 bozo 的同事的链接和 /cgi-bin 中的虚假脚本，你可以指定： 

wget -I /people,/cgi-bin http://host/people/bozo/
‘-X list’ ¶
‘--exclude list’
‘exclude_directories = list’

‘-X’ 选项与 ‘-I’ 完全相反——这是一个从下载中 排除 的目录列表。 例如。 如果你不希望 Wget 从 /cgi-bin 目录下载东西，可以在命令行上指定 ‘-X /cgi-bin’。

与 ‘-A’/‘-R’ 相同，这两个选项可以组合使用，以便更好地微调下载子目录。 例如。 如果你想从 /pub 层次结构中加载所有文件，除了 /pub/worthless，可以指定 ‘-I/pub -X/pub/worthless’。

‘-np’ ¶
‘--no-parent’
‘no_parent = on’

限制目录的最简单且通常非常有用的方法是禁止检索指向起始目录 以上 层次的链接，即禁止上升到父目录/目录。

在这种情况下，‘--no-parent’ 选项（简称 ‘-np’）非常有用。 使用它可以保证你永远不会离开现有的层次结构。 假设你用 Wget 发出： 

wget -r --no-parent http://somehost/~luzer/my-archive/

你可以放心，不会跟随任何指向 /~his-girls-homepage/ 或 /~luzer/all-my-mpegs/ 的引用。 只有你感兴趣的存档会被下载。 本质上，‘--no-parent’ 类似于 ‘-I/~luzer/my-archive’，只是它以更智能的方式处理重定向。

注意，对于 HTTP（和 HTTPS），尾随斜杠对 ‘--no-parent’ 非常重要。 HTTP 没有“目录”的概念——Wget 依赖于你来指示什么是目录，什么不是。 在 ‘http://foo/bar/’ 中，Wget 会将 ‘bar’ 视为目录，而在 ‘http://foo/bar’（没有尾随斜杠）中，‘bar’ 将被视为文件名（因此 ‘--no-parent’ 将毫无意义，因为它的父级是 ‘/’）。

4.4 相对链接 ¶

当 ‘-L’ 打开时，只有相对链接会被跟随。 相对链接在这里被定义为不指向网络服务器根目录的链接。 例如，这些链接是相对的： 

<a href="foo.gif">
<a href="foo/bar.gif">
<a href="../foo/bar.gif">

这些链接不是相对的： 

<a href="/foo.gif">
<a href="/foo/bar.gif">
<a href="http://www.example.com/foo/bar.gif">

使用这个选项可以保证递归获取不会跨越主机，即使没有 ‘-H’ 也一样。 在简单的情况下，它也允许下载“只需工作”而无需转换链接。

这个选项可能并不很有用，并可能在未来的版本中被移除。

4.5 跟随FTP链接 ¶

对于 FTP 的规则有点特殊，因为对于它们来说是必要的。 FTP 链接在 HTML 文档中常常被包含以供参考，并且默认情况下下载它们常常是不方便的。

要从 HTML 文档中跟随 FTP 链接，你需要指定 ‘--follow-ftp’ 选项。 做完这个后，FTP 链接将会跨越主机，无论 ‘-H’ 设置如何。 这是合理的， 因为 FTP 链接很少指向与 HTTP 服务器所在的同一主机。 出于类似的原因， ‘-L’ 选项对这种下载没有效果。 另一方面，域名接受 (‘-D’) 和后缀规则 (‘-A’ 和 ‘-R’) 应用正常。

还要注意的是，跟随的链接到 FTP 目录将不会被递归地进一步获取。

5 时间戳 ¶

从互联网镜像信息的最重要方面之一是更新你的归档。

一次又一次地下载整个归档，只是为了替换少量更改的文件，这在带宽和金钱上都是很费的，同时也需要更新的时间。 这就是为什么所有镜像工具 都提供增量更新的选项。

这种更新机制意味着远程服务器会被扫描以寻找 新的 文件。 只有那些新文件会被下载来 替换旧的文件。

如果满足以下两个条件之一，一个文件就被认为是新的：

  1. 本地不存在该名称的文件。
  2. 本地存在该名称的文件，但远程文件在本地文件之后被修改。

为了实现这一点，程序需要知道本地和远程文件的最后修改时间。 我们称这个信息为文件的 时间戳。

GNU Wget 中的时间戳通过使用 ‘--timestamping’ (‘-N’) 选项或通过 timestamping = on 指令在 .wgetrc 中打开。 使用这个选项，对于每个它想要下载的文件， Wget 将会检查是否存在同名的本地文件。 如果存在，并且远程文件不是更新的，Wget 将不会下载它。

如果本地文件不存在，或文件的大小不匹配，Wget 将不管时间戳如何都会下载远程文件。

5.1 时间戳使用 ¶

时间戳的使用很简单。 假设你想下载一个文件，以便它保持其修改日期。 

wget -S http://www.gnu.ai.mit.edu/

一个简单的ls -l显示本地文件上的时间戳等于服务器返回的Last-Modified头的状态。 如你所见，时间戳信息在本地被保留，即使没有‘-N’（至少对于HTTP）。

几天后，你想让Wget检查远程文件是否已更改，如果已更改则下载它。 

wget -N http://www.gnu.ai.mit.edu/

Wget将向服务器询问最后修改日期。 如果本地文件的时间戳与服务器相同或更新，则不会重新获取远程文件。 但是，如果远程文件更新，Wget将继续获取它。

对于FTP也是如此。 例如： 

wget "ftp://ftp.ifi.uio.no/pub/emacs/gnus/*"

（该URL周围的引号是为了防止shell尝试解释‘*’。）

下载后，本地目录列表将显示时间戳与远程服务器上的时间戳匹配。 使用‘-N’重新发出命令将使Wget仅重新获取自上次下载以来已修改的文件。

如果你想每周镜像GNU存档，你可以每周使用如下命令： 

wget --timestamping -r ftp://ftp.gnu.org/pub/gnu/

请注意，时间戳仅适用于服务器提供时间戳的文件。 对于HTTP，这取决于获取Last-Modified头。 对于FTP，这取决于获取一个日期格式的目录列表，Wget可以解析该格式（参见FTP时间戳内部）。

5.2 HTTP时间戳内部 ¶

HTTP中的时间戳是通过检查Last-Modified头来实现的。 如果你想通过HTTP检索文件foo.html，Wget将检查foo.html是否在本地存在。 如果不存在，foo.html将被无条件检索。

如果文件确实存在于本地，Wget将首先检查其本地时间戳（类似于ls -l检查它的方式），然后向远程服务器发送HEAD请求，要求远程文件的信息。

检查Last-Modified头以找出哪个文件最近被修改（使其“更新”）。 如果远程文件更新，它将被下载；如果它更旧，Wget将放弃。2

当‘--backup-converted’（‘-K’）与‘-N’一起指定时，服务器文件‘X’将与本地文件‘X.orig’进行比较，如果存在，而不是与本地文件‘X’进行比较，如果它已被‘--convert-links’（‘-k’）转换，则总是不同。

可以说，HTTP时间戳应该使用If-Modified-Since请求来实现。

5.3 FTP时间戳内部 ¶

理论上，FTP时间戳的工作方式与HTTP大致相同，只是 FTP没有头部——时间戳必须从目录 列表中提取出来。

如果FTP下载是递归的或使用通配符，Wget将使用 FTP LIST命令获取包含所需文件的目录的 文件列表。 它将尝试分析列表， 将其视为Unix ls -l输出，提取时间戳。 其余部分与HTTP完全相同。 请注意，当 从FTP服务器检索单个文件而不使用 通配符或递归时，除非指定了‘-N’，否则不会下载列表文件（因此 文件不会被时间戳标记）。

假设每个目录列表都是Unix风格的列表可能 听起来极其限制，但实际上并非如此，因为许多 非Unix FTP服务器使用Unixoid列表格式，因为大多数 （所有？）客户端都理解它。 请记住，RFC959 没有定义获取文件列表的标准方式，更不用说时间戳了。 我们只能希望未来的标准会定义这一点。

另一个非标准解决方案包括使用MDTM命令 ，该命令由一些FTP服务器支持（包括流行的 wu-ftpd），它返回指定文件的确切时间。 Wget将来可能会支持此命令。

7.1 简单用法 ¶

  • 假设你想下载一个URL。 只需输入： 
    wget http://fly.srk.fer.hr/
  • 但是如果连接速度慢，文件又长，会发生什么？ 连接可能会在文件完全检索之前失败，不止一次。 在这种情况下，Wget将尝试获取文件，直到它要么获取整个文件，要么超过默认的重试次数（这是20次）。 将尝试次数更改为45次很容易，以确保整个文件安全到达： 
    wget --tries=45 http://fly.srk.fer.hr/jpg/flyweb.jpg
  • 现在让我们让Wget在后台工作，并将其进度写入日志文件log。 输入‘--tries’很累，所以我们 将使用‘-t’。 
    wget -t 45 -o log http://fly.srk.fer.hr/jpg/flyweb.jpg &

    行末的&确保Wget在后台工作。 要无限次重试，请使用‘-t inf’。

  • FTP的使用同样简单。 Wget会处理登录和密码。 
    wget ftp://gnjilux.srk.fer.hr/welcome.msg
  • 如果你指定一个目录，Wget将检索目录列表，解析它并将其转换为HTML。 尝试： 
    wget ftp://ftp.gnu.org/pub/gnu/
links index.html

7.2 高级用法 ¶

  • 你有一个包含你想要下载的URL的文件吗？ 使用 ‘-i’开关： 
    wget -i file

    如果你指定‘-’作为文件名，URLs将从 标准输入读取。

  • 创建GNU网站的五层深度镜像图像，与原始网站具有相同的目录结构，每个文档只尝试一次，将活动日志保存到gnulog： 
    wget -r https://www.gnu.org/ -o gnulog
  • 与上述相同，但将下载文件中的链接转换为指向本地文件，以便你可以离线查看文档： 
    wget --convert-links -r https://www.gnu.org/ -o gnulog
  • 仅检索一个HTML页面，但确保页面显示所需的所有元素，如内联图像和外部样式表，也被下载。 还要确保下载的页面引用下载的链接。 
    wget -p --convert-links http://www.example.com/dir/page.html

    HTML页面将保存到www.example.com/dir/page.html， 图像、样式表等将根据它们在远程服务器上的位置保存到www.example.com/下的某个位置。

  • 与上述相同，但没有www.example.com/目录。 实际上，我不想要所有这些随机的服务器目录——只需将所有这些文件保存到当前目录下的download/子目录中。 
    wget -p --convert-links -nH -nd -Pdownload \
     http://www.example.com/dir/page.html
  • 检索‘www.lycos.com’的index.html，显示原始服务器头： 
    wget -S http://www.lycos.com/
  • 将服务器头与文件一起保存，可能用于后处理。 
    wget --save-headers http://www.lycos.com/
more index.html
  • 检索‘wuarchive.wustl.edu’的前两层，将它们保存到/tmp。 
    wget -r -l2 -P/tmp ftp://wuarchive.wustl.edu/
  • 你想从HTTP服务器上的目录下载所有GIF。 你尝试了‘wget http://www.example.com/dir/*.gif’，但那 没有工作，因为HTTP检索不支持通配符。 在 这种情况下，使用： 
    wget -r -l1 --no-parent -A.gif http://www.example.com/dir/

    更详细，但效果相同。 ‘-r -l1’意味着 递归检索（参见递归下载），最大深度为1。 ‘--no-parent’意味着忽略对父目录的引用（参见基于目录的限制），而‘-A.gif’意味着只下载GIF文件。 ‘-A "*.gif"’也会 工作。

  • 假设你在下载过程中，Wget被中断了。 现在你不想覆盖已经存在的文件。 它将是： 
    wget -nc -r https://www.gnu.org/
  • 如果你想编码自己的用户名和密码到HTTP或 FTP，使用适当的URL语法（参见URL格式）。 
    wget ftp://hniksic:mypassword@unix.example.com/.emacs

    请注意，这种用法在多用户系统上不建议使用，因为它会向查看ps输出的人泄露您的密码。

  • 您希望输出文档发送到标准输出而不是文件吗？ 
    wget -O - http://jagor.srce.hr/ http://www.srce.hr/

    您还可以结合这两个选项，并通过管道从远程热点列表检索文档： 

    wget -O - http://cool.list.com/ | wget --force-html -i -

7.3 非常高级的用法 ¶

  • 如果您希望Wget保持页面（或FTP子目录）的镜像，请使用‘--mirror’（‘-m’），这是‘-r -l inf -N’的简写。 您可以将Wget放入crontab文件中，要求它每周日重新检查一个站点： 
    crontab
0 0 * * 0 wget --mirror https://www.gnu.org/ -o /home/me/weeklog
  • 除了上述内容外，您还希望链接被转换为本地查看。 但是，在阅读了本手册后，您知道链接转换与时间戳处理不太兼容，因此您还希望Wget在转换前备份原始的HTML文件。 Wget的调用将如下所示： 
    wget --mirror --convert-links --backup-converted  \
     https://www.gnu.org/ -o /home/me/weeklog
  • 但您也注意到，当HTML文件以‘.html’以外的扩展名保存时，本地查看效果不佳，可能是因为它们以index.cgi的形式提供。 因此，您希望Wget将所有以内容类型‘text/html’或‘application/xhtml+xml’提供的文件重命名为name.html。 
    wget --mirror --convert-links --backup-converted \
     --adjust-extension -o /home/me/weeklog      \
     https://www.gnu.org/

    或者，输入更少： 

    wget -m -k -K -E https://www.gnu.org/ -o /home/me/weeklog

8.1 代理 ¶

代理是专为将数据从远程服务器传输到本地客户端而设计的特殊用途HTTP服务器。 代理的一个典型用途是减轻慢速连接用户的网络负载。 这是通过将所有HTTP和FTP请求通过代理进行，代理缓存传输的数据来实现的。 当再次请求缓存的资源时，代理将从缓存中返回数据。 代理的另一个用途是用于那些出于安全原因将其内部网络与互联网其他部分隔离的公司。 为了从网络获取信息，他们的用户通过授权的代理连接并检索远程数据。

Wget支持HTTP和FTP检索的代理。 指定代理位置的标准方式，Wget识别的方式，是使用以下环境变量：

http_proxy
https_proxy

如果设置了，http_proxy和https_proxy变量应分别包含HTTP和HTTPS连接的代理的URL。

ftp_proxy

该变量应包含用于FTP连接的代理的URL。 通常，http_proxy和ftp_proxy被设置为相同的URL。

no_proxy

该变量应包含一个逗号分隔的域名扩展列表，代理不应用于这些扩展。 例如，如果no_proxy的值为‘.mit.edu’，则不会使用代理从MIT检索文档。

除了环境变量外，代理位置和设置也可以在Wget内部指定。

‘--no-proxy’
‘proxy = on/off’

此选项和相应的命令可用于抑制代理的使用，即使设置了适当的环境变量。

‘http_proxy = URL’
‘https_proxy = URL’
‘ftp_proxy = URL’
‘no_proxy = string’

这些启动文件变量允许您覆盖环境指定的代理设置。

一些代理服务器需要授权才能使用它们。 授权由用户名和密码组成，必须由Wget发送。 与HTTP授权一样，存在几种认证方案。 对于代理授权，目前仅实现了Basic认证方案。

http://hniksic:mypassword@proxy.company.com:8001/

或者，您可以使用‘proxy-user’和‘proxy-password’选项，以及等效的.wgetrc设置proxy_user和proxy_password来设置代理用户名和密码。

在提交错误报告之前，请尝试遵循一些 简单的指导原则。

  1. 请尝试确认你看到的行为确实是一个错误。 如果 Wget崩溃，这是一个错误。 如果Wget的行为与文档不符， 这是一个错误。 如果事情运作奇怪，但你不确定它们应该如何运作，这很可能是一个错误，但你可能想要再次检查文档和邮件列表（参见邮件列表）。
  2. 尝试在尽可能简单的情况下重复这个错误。 例如。 如果 Wget在下载‘wget -rl0 -kKE -t5 --no-proxy http://example.com -o /tmp/log’时崩溃，你应该尝试看看崩溃是否可以重复，以及是否会在使用更简单的选项集时发生。 你甚至可能尝试从崩溃发生的页面开始下载，看看该页面是否以某种方式触发了崩溃。

    另外，虽然我可能想知道你的 .wgetrc文件的内容，但仅仅将其转储到调试信息中可能是一个坏主意。 相反，你应该首先尝试看看如果移动.wgetrc文件，错误是否还会重复。 只有当发现 .wgetrc设置影响错误时，才将文件的相关部分邮寄给我。

  3. 请使用‘-d’选项启动Wget，并将结果输出（或相关部分）发送给我们。 如果Wget是在没有调试支持的情况下编译的，请重新编译它——开启调试支持后追踪错误要容易得多。

    注意：在将调试日志发送到错误地址之前，请确保删除任何潜在的敏感信息。 -d不会特意收集敏感信息，但日志会包含Wget与服务器通信的相当完整的记录，这可能包括密码和下载的数据片段。 由于错误地址是公开存档的，你可以假设所有错误报告对公众都是可见的。

8.8 信号 ¶

由于Wget的目的是后台工作，它会捕获挂起信号（SIGHUP）并忽略它。 如果输出是在标准 输出上，它将被重定向到一个名为wget-log的文件。 否则，SIGHUP将被忽略。 这在你希望 在启动Wget后重定向其输出时非常方便。 

$ wget http://www.gnus.org/dist/gnus.tar.gz &
……
$ kill -HUP %%
SIGHUP received, redirecting output to `wget-log'.

除此之外，Wget不会以任何方式干扰信号。 C-c、kill -TERM和kill -KILL应该同样杀死它。

9.1 机器人排除 ¶

让Wget在网站上漫无目的地游荡，吸收所有可用的数据，这是极其容易的。 ‘wget -r site’, 你就设置好了。 很棒？ 对服务器管理员来说不是。

只要Wget只检索静态页面，并且以合理的速率进行（参见‘--wait’选项），问题不大。 问题是Wget无法区分最小的静态页面和最苛刻的CGI。 我知道的一个网站有一个由CGI Perl脚本处理的部分，该脚本将Info文件即时转换为HTML。 这个脚本很慢，但对于查看偶尔的Info文件的人类用户来说足够好。 然而，当某人的递归Wget下载偶然发现通过脚本链接到所有Info文件的索引页面时，系统会被拖垮，而不会为用户提供任何有用的东西（这个转换Info文件的任务可以在本地完成，并且可以通过info命令访问系统上所有已安装的GNU软件的Info文档）。

为了避免这种事故，以及保护需要防止行为良好的机器人访问的文档的隐私，发明了机器人排除的概念。 这个想法是，服务器管理员和文档作者可以指定他们希望保护免受机器人访问的网站部分以及他们允许访问的部分。

最流行的机制，以及所有主要机器人支持的事实标准，是Martijn Koster等人在1994年编写的“机器人排除标准”（RES）。 它指定了一个文本文件的格式，该文件包含指示机器人避免哪些URL路径的指令。 为了让机器人找到，这些规范必须放在服务器根目录的/robots.txt中，机器人会下载并解析它。

尽管Wget不是严格意义上的网络机器人，但它可以在用户不干预下载单个页面的情况下下载网站的大部分内容。 因此，Wget在递归下载时遵守RES。 例如，当你发出： 

wget -r http://www.example.com/

首先会下载‘www.example.com’的索引。 如果Wget发现它想从该服务器下载更多文档，它将请求‘http://www.example.com/robots.txt’，如果找到，将使用它进行进一步下载。 robots.txt每个服务器只加载一次。

直到1.8版本，Wget支持Martijn Koster在1994年编写的标准的第一个版本，可在http://www.robotstxt.org/orig.html找到。 从1.8版本开始，Wget支持互联网草案‘<draft-koster-robots-00.txt>’中指定的额外指令，标题为“网络机器人控制方法”。 据我所知，该草案从未成为RFC，可在http://www.robotstxt.org/norobots-rfc.txt找到。

本手册不再包含机器人排除标准的文本。

第二个不太为人所知的机制，使单个文档的作者能够指定他们是否希望机器人跟随文件中的链接。 这是通过使用META标签实现的，像这样： 

<meta name="robots" content="nofollow">

这在http://www.robotstxt.org/meta.html有详细解释。 Wget除了通常的/robots.txt排除外，还支持这种机器人排除方法。

如果你知道你在做什么，并且真的希望关闭机器人排除，可以在你的.wgetrc中将robots变量设置为‘off’。 你可以通过命令行使用-e开关达到同样的效果，例如‘wget -e robots=off url...’.

9.2 安全考虑 ¶

使用Wget时，你必须意识到它会通过网络发送未加密的密码，这可能会带来安全问题。 以下是主要问题及一些解决方案。

  1. 命令行上的密码使用ps可见。 最好的解决方法是使用wget -i -并将URLs输入到Wget的标准输入中，每个URL单独一行，以C-d结束。 另一种解决方法是使用.netrc来存储密码；然而，存储未加密的密码也被认为是一种安全风险。
  2. 使用不安全的basic认证方案，未加密的密码通过网络路由器和网关传输。
  3. FTP密码也完全没有加密。 目前对此没有好的解决方案。
  4. 尽管Wget的“正常”输出试图隐藏密码，调试日志会以各种形式显示它们。 通过发送调试日志时小心（是的，即使你发送给我）可以避免这个问题。

    “次要部分”是指文档的命名附录或前言部分，专门处理文档的出版商或作者与文档整体主题（或相关事项）的关系，并且不包含任何可以直接属于该整体主题的内容。 （因此，如果文档部分是一本数学教科书，次要部分可能不解释任何数学。） 这种关系可能是与主题或相关事项的历史联系，或者是关于它们的法律、商业、哲学、伦理或政治立场。

    “不变章节”是某些次要章节，其标题被指定为不变章节的标题，在声明文档根据此许可证发布的公告中。 如果一个章节不符合上述次要章节的定义，则不允许将其指定为不变。 文档可能包含零个不变章节。 如果文档没有标识任何不变章节，则没有。

    “封面文本”是某些短文本段落，在声明文档根据此许可证发布的公告中列为封面文本或封底文本。 封面文本最多5个字，封底文本最多25个字。

    文档的“透明”副本意味着机器可读的副本，以规范公开可用的格式表示，适合用通用文本编辑器直接修订文档（对于由像素组成的图像）通用绘画程序或（对于绘图）一些广泛可用的绘图编辑器，并且适合输入到文本格式化程序或自动翻译成适合输入到文本格式化程序的各种格式。 以其他方式透明的文件格式制作的副本，其标记或标记的缺失已被安排以阻碍或阻止读者进行后续修改，不是透明的。 如果用于任何大量文本，图像格式不是透明的。 不是“透明”的副本称为“不透明”；。

    适合透明副本的格式示例包括纯ASCII无标记、Texinfo输入格式、LaTeX输入格式、使用公开可用DTD的SGML或XML，以及符合标准的简单HTML、PostScript或PDF设计用于人工修改。 透明图像格式的示例包括PNG、XCF和JPG。 不透明格式包括只能由专有文字处理器读取和编辑的专有格式，DTD和/或处理工具通常不可用的SGML或XML，以及一些文字处理器仅为输出目的生成的机器生成的HTML、PostScript或PDF。

    “标题页”对于印刷书籍来说，是指标题页本身，以及为清晰显示本许可证要求在标题页中出现的材料所需的后续页面。 对于没有标题页的作品，“标题页”是指作品标题最显眼出现前的文本，紧接正文开始之前。

    “出版商”是指向公众分发文档副本的任何个人或实体。

    “标题为XYZ”的部分是指文档的命名子单元，其标题要么是精确的XYZ，要么在翻译XYZ为另一种语言的文本后包含XYZ。 （这里XYZ代表下面提到的特定章节名称，如“致谢”、“献词”、“背书”或“历史”。） 当您修改文档时，“保留标题”意味着根据此定义，它仍然是一个“标题为XYZ”的部分。

    文档可能包括与本许可证适用于文档的声明并列的免责声明。 这些免责声明被认为是通过引用包含在本许可证中，但仅限于免责声明：这些免责声明可能具有的任何其他含义是无效的，并且对本许可证的含义没有影响。

  3. 逐字复制

    您可以在任何媒介上复制和分发文档，无论是商业还是非商业用途，前提是本许可证、版权声明和声明本许可证适用于文档的许可证声明在所有副本中复制，并且您不向本许可证的条件添加任何其他条件。 您不得使用技术措施来阻碍或控制您制作或分发的副本的阅读或进一步复制。 但是，您可以接受补偿以换取副本。 如果您分发足够数量的副本，您还必须遵守第3节中的条件。

    您也可以在上述相同条件下出借副本，并且您可以公开展示副本。

  4. 大量复制

    如果您发布文档的印刷副本（或通常有印刷封面的媒介中的副本），数量超过100份，并且文档的许可证声明需要封面文本，您必须将这些副本装入封面，清晰易读地携带所有这些封面文本：封面上的封面文本和封底上的封底文本。 两个封面还必须清晰易读地标识您为这些副本的出版商。 封面必须呈现完整标题，标题的所有单词同样突出和可见。 您可以在封面上添加其他材料。 只要封面上的更改保留了文档的标题并满足这些条件，就可以在其他方面视为逐字复制。

    如果任一封面所需的文本太多以至于无法清晰放置，您应该将列出的第一个文本（尽可能合理地适合）放在实际封面上，并将其余的继续到相邻页面。

    如果您发布或分发超过100份文档的不透明副本，您必须要么包括每份不透明副本的机器可读透明副本，要么在每份不透明副本中或与之一起声明一个计算机网络位置，从该位置，一般网络使用公众可以使用公共标准网络协议下载文档的完整透明副本，无添加材料。 如果您选择后者，您必须在开始大量分发不透明副本时采取合理谨慎的步骤，以确保此透明副本在声明的位置保持可访问，直到至少一年后您最后一次向公众分发该版本的不透明副本（直接或通过您的代理商或零售商）。

    建议但不要求，在重新分发大量副本之前，您应联系文档的作者，以便他们有机会为您提供文档的更新版本。

  5. 修改

    您可以在上述第2和第3节的条件下复制和分发文档的修改版本，前提是您按照本许可证的条款发布修改版本，修改版本承担文档的角色，从而许可分发和修改修改版本给任何拥有其副本的人。 此外，您必须在修改版本中做以下事情：

    1. 在标题页（以及封面，如果有的话）使用与文档不同的标题，以及与之前版本不同的标题（如果有的话，应在文档的历史部分列出）。 如果该版本的原始出版商给予许可，您可以使用与之前版本相同的标题。
    2. 在标题页上列出作为作者的一个或多个人或实体，负责修改版本中的修改，以及至少五位文档的主要作者（如果主要作者少于五位，则列出所有主要作者），除非他们免除您这一要求。
    3. 在标题页上声明修改版本的出版商名称，作为出版商。
    4. 保留文档的所有版权声明。
    5. 在您的修改旁边添加适当的版权声明，与其他版权声明相邻。
    6. 在版权声明之后立即包含一个许可证声明，以本许可证的条款授予公众使用修改版本的许可，形式如下面的附录所示。
    7. 在该许可证声明中保留文档许可证声明中给出的不变章节和必需封面文本的完整列表。
    8. 包含本许可证的未更改副本。
    9. 保留标题为“历史”的部分，保留其标题，并在其中添加一个项目，至少说明标题页上给出的修改版本的标题、年份、新作者和出版商。 如果文档中没有标题为“历史”的部分，则创建一个部分，说明标题页上给出的文档的标题、年份、作者和出版商，然后添加一个项目，描述如前一句所述的修改版本。
    10. 保留文档中给出的网络位置（如果有的话），以便公众访问文档的透明副本，同样保留文档中给出的基于它的之前版本的网络位置。 这些可以放在“历史”部分。 对于在文档本身发布至少四年前出版的作品，或者如果它所指版本的原始出版商给予许可，您可以省略网络位置。
    11. 对于任何标题为“致谢”或“献词”的部分，保留该部分的标题，并保留该部分中所有贡献者致谢和/或献词的内容和语气。
    12. 保留文档的所有不变章节，其文本和标题均未更改。 章节编号或等效内容不被视为章节标题的一部分。
    13. 删除任何标题为“背书”的部分；。 这样的部分可能不包括在修改版本中。
    14. 不要将任何现有部分重新命名为“背书”或与任何不变章节的标题冲突。
    15. 保留所有免责声明。

    如果修改版本包括新的前言部分或附录，这些部分符合次要章节的资格且不包含从文档复制的材料，您可以选择将其中一些或全部章节指定为不变。 为此，将它们的标题添加到修改版本的许可证声明中的不变章节列表中。 这些标题必须与任何其他章节标题不同。

    您可以添加一个标题为“背书”的部分，前提是它仅包含各方对您修改版本的背书——例如，同行评审的声明或文本已被组织批准作为标准的权威定义。

    您可以在修改版本的封面文本列表末尾添加最多五个单词的前封面文本和最多25个单词的后封面文本。 只有一段前封面文本和一段后封面文本可以由（或通过安排由）任何实体添加。 如果文档已经包括由您或您代表的同一实体通过安排添加的相同封面的封面文本，您可能不能添加另一个；但您可以在明确获得添加旧文本的先前出版商的许可后替换旧文本。

    文档的作者和出版商不通过本许可证授予使用他们的名字进行宣传或断言或暗示对任何修改版本的背书的许可。

  6. 合并文档

    您可以将文档与本许可证下发布的其他文档合并，按照上述第4节为修改版本定义的条款，前提是您在合并中包含所有原始文档的所有不变章节，未修改，并在其许可证声明中将它们全部列为合并作品的不变章节，并且您保留所有它们的免责声明。

    合并作品只需包含本许可证的一个副本，多个相同的不变章节可以用一个副本替换。 如果有多个具有相同名称但内容不同的不变章节，通过在每章的标题末尾添加括号中的原始作者或出版商名称（如果已知）或唯一编号，使每章的标题唯一。 对合并作品许可证声明中不变章节列表中的章节标题进行相同的调整。

    在组合中，您必须将各个原始文档中标题为“历史”的部分组合在一起，形成一个标题为“历史”的部分；同样地，将标题为“致谢”的部分和标题为“献词”的部分组合在一起。 您必须删除所有标题为“背书”的部分。

  7. 文档集合

    您可以制作一个包含本文档和根据本许可证发布的其他文档的集合，并用集合中包含的单个副本替换各个文档中的本许可证副本，前提是您遵循本许可证的规则，在所有其他方面对每个文档进行逐字复制。

    您可以从这样的集合中提取单个文档，并根据本许可证单独分发它，前提是您将本许可证的副本插入到提取的文档中，并在所有其他方面遵循本许可证关于该文档的逐字复制。

  8. 与独立作品的聚合

    如果由于编译而产生的版权不被用于限制编译用户的合法权利超出单个作品允许的范围，那么将本文档或其衍生作品与其他独立和独立的文档或作品在存储或分发介质的卷上或卷内的编译称为“聚合”。 当本文档包含在聚合中时，本许可证不适用于聚合中不是本文档衍生作品的其他作品。

    如果第3节的封面文本要求适用于本文档的这些副本，那么如果本文档少于整个聚合的一半，本文档的封面文本可以放置在聚合中包围本文档的封面上，或者如果本文档是电子形式的，则放置在封面的电子等效物上。 否则，它们必须出现在包围整个聚合的印刷封面上。

  9. 翻译

    翻译被视为一种修改，因此您可以根据第4节的条款分发本文档的翻译。 用翻译替换不变部分需要其版权持有者的特别许可，但您可以在这些不变部分的原始版本之外，包括部分或全部不变部分的翻译。 您可以包括本许可证的翻译，以及文档中的所有许可证声明和任何免责声明，前提是您还包括本许可证的原始英文版本以及这些声明和免责声明的原始版本。 如果翻译与本许可证或声明或免责声明的原始版本之间存在分歧，原始版本将优先。

    如果文档中的部分标题为“致谢”、“献词”或“历史”，则保留其标题（第1节）的要求（第4节）通常需要更改实际标题。

  10. 终止

    除非本许可证明确允许，否则您不得复制、修改、再许可或分发本文档。 任何尝试以其他方式复制、修改、再许可或分发它的行为都是无效的，并将自动终止您在本许可证下的权利。

    然而，如果您停止所有违反本许可证的行为，那么您从特定版权持有者那里获得的许可证将被恢复（a）暂时恢复，除非并且直到版权持有者明确且最终终止您的许可证，以及（b）永久恢复，如果版权持有者在停止后60天内未能通过某种合理的方式通知您违规行为。

    根据本节终止您的权利不会终止根据本许可证从您那里收到副本或权利的方的许可证。 如果您的权利已被终止且未永久恢复，收到部分或全部相同材料的副本不会给您任何使用它的权利。

  11. 本许可证的未来修订

    自由软件基金会可能会不时发布GNU自由文档许可证的新修订版本。 这些新版本在精神上将与当前版本相似，但可能在细节上有所不同，以解决新问题或关注点。 参见 http://www.gnu.org/copyleft/。

    每个版本的许可证都有一个独特的版本号。 如果文档指定了本许可证的特定编号版本“或任何后续版本”适用于它，您可以选择遵循该指定版本或自由软件基金会发布的任何后续版本（非草案）的条款和条件。 如果文档没有指定本许可证的版本号，您可以选择自由软件基金会发布的任何版本（非草案）。 如果文档指定了代理可以决定可以使用本许可证的哪些未来版本，则该代理对版本的公开接受声明永久授权您为该文档选择该版本。

  12. 再许可

    “大规模多作者协作站点”（或“MMC站点”）是指发布可版权作品并提供显著设施供任何人编辑这些作品的任何万维网服务器。 一个任何人都可以编辑的公共wiki就是这样一个服务器的例子。 站点中包含的“大规模多作者协作”（或“MMC”）是指在MMC站点上发布的任何一组可版权作品。

    “CC-BY-SA”是指由Creative Commons Corporation发布的Creative Commons Attribution-Share Alike 3.0许可证，这是一家位于加利福尼亚州旧金山的非营利公司，以及该组织发布的该许可证的未来copyleft版本。

    “合并”是指将文档的全部或部分作为另一文档的一部分发布或重新发布。

    如果MMC是根据本许可证授权的，并且所有首次在本许可证下发布的作品（除了本MMC之外）随后全部或部分并入MMC，（1）没有封面文本或不变部分，（2）因此并入日期在2008年11月1日之前，则该MMC“有资格重新授权”。

    MMC站点的运营商可以在2009年8月1日之前的任何时间，在相同站点上以CC-BY-SA重新发布包含在站点中的MMC，前提是该MMC有资格重新授权。
